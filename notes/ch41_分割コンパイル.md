# 第41章: 分割コンパイル

## 要点

### ビルドの全体像

C++ プログラムができるまでの流れ：

```
ソースファイル (.cpp)
       │
   [プリプロセッサ]   ← #include, #define を展開
       │
   展開済みソース
       │
   [コンパイラ]      ← 構文解析、機械語に変換
       │
   オブジェクトファイル (.o / .obj)
       │
   [リンカ]          ← 複数の .o をまとめる、外部参照を解決
       │
   実行ファイル (.exe / a.out)
```

### ソースファイルとオブジェクトファイル

| | ソースファイル | オブジェクトファイル |
|---|---|---|
| 何 | 人が書くテキスト | コンパイラ出力のバイナリ |
| 拡張子 | `.cpp` / `.h` / `.hpp` | `.o`（GCC/Clang）／ `.obj`（MSVC） |
| 内容 | C++ コード | 機械語 + シンボル情報 |
| 役割 | プログラムの設計 | 中間生成物 |
| 単独で動く？ | ❌（コンパイル必要） | ❌（リンク必要） |
| 編集可能？ | ✅ | ❌（バイナリ） |

### 段階的にビルドしてみる

```bash
# 一括ビルド（普段の書き方）
g++ main.cpp add.cpp -o app

# 段階的ビルド（内部ではこれと同じことをしている）
g++ -c main.cpp        # main.o ができる
g++ -c add.cpp         # add.o ができる
g++ main.o add.o -o app  # リンクして実行ファイル
```

`-c` は「コンパイルだけしてリンクしない」オプション。

### オブジェクトファイルの中身

機械語に変換済みだが、**他のファイルにある関数の実際のアドレスはまだ埋まっていない**。

```
[コードセクション]
  main:
    call add               ← add の場所はまだ不明（未解決）

[シンボルテーブル]
  main: 定義済み
  add: 未解決（外部から探す）
```

`nm` コマンドで確認できる：

```
$ nm main.o
                 U _Z3addii   ← U = Undefined（未解決）
0000000000000000 T main       ← T = Text（定義済み）
```

### 分割コンパイルの利点

#### ① 増分ビルドが速い

```
main.cpp を1行直した
  → main.o だけ作り直す
  → add.o, sub.o は再利用
```

大規模プロジェクトでは「変更したファイルだけ再コンパイル」が必須。

#### ② 並列ビルド

複数の `.cpp` を別 CPU コアで同時にコンパイル可能。

```bash
make -j8     # 8並列でビルド
```

#### ③ ライブラリ化

複数の `.o` をまとめて静的ライブラリ（`.a` / `.lib`）や動的ライブラリ（`.so` / `.dll`）にできる。

### 変数の宣言と定義 — `extern`

関数と同じく、変数も **宣言と定義を分けられる**。

```cpp
// 関数：分けるのは普通
void foo();              // 宣言
void foo() { /* ... */ } // 定義

// 変数：default は「宣言＝定義」
int x = 0;       // これは「定義」（実体が作られる）
extern int x;    // 宣言（実体は別の .cpp）
```

`extern` の意味：**「この変数は別の翻訳単位で定義されている。ここでは使わせて」** とコンパイラに伝える。実体は作らない。

### 典型パターン：複数の `.cpp` で変数を共有

```cpp
// counter.hpp
#pragma once
extern int counter;          // 宣言だけ
```

```cpp
// counter.cpp
int counter = 0;             // ここで定義（1箇所だけ）
```

```cpp
// main.cpp
#include "counter.hpp"
int main() { counter = 10; } // counter.cpp の counter を使う
```

```cpp
// other.cpp
#include "counter.hpp"
void foo() { counter++; }    // 同じ counter
```

3つの `.cpp` が **同じ `counter`** を共有する。

### なぜ `extern` が必要か

ヘッダに直接 `int counter = 0;` と書くと **多重定義エラー** になる。

```cpp
// counter.hpp
int counter = 0;             // ❌ ヘッダに書くと…
// → main.cpp と other.cpp の両方で定義される → リンクエラー
```

`#pragma once` は同じ翻訳単位での多重 include を防ぐが、**異なる翻訳単位への展開** は防げない。

### ODR（One Definition Rule）

> **同じ名前の関数や変数の定義は、プログラム全体で1つだけ**

普通の定義は **strong symbol** として `.o` に記録され、リンク時に重複していたらエラー。

| シンボル種別 | 重複時の挙動 |
|---|---|
| **strong**（普通の定義） | 重複したらエラー |
| **weak**（`inline` 由来） | 重複してもエラーにならない、1つだけ採用 |

### `inline` キーワード — ODR の例外

`inline` を付けると **「複数の翻訳単位に同じ定義が現れてもよい」** とコンパイラ・リンカに保証する宣言になる。

```cpp
// foo.hpp
inline int square(int x) { return x * x; }   // ヘッダに本体OK

inline int counter = 0;                       // C++17 inline 変数
```

仕組み：`inline` のシンボルは **weak symbol** としてマークされ、リンカが「全部同じ意味だから1つに統合」と判断する。

```
$ nm a.o
0000000000000000 W counter   ← W = weak（inline 由来）
```

### `inline` の本来の意味の変遷

歴史的には「**関数を呼び出さずに展開してね**」という最適化のヒント。
現代では **「ODR の例外を作る指定」** が本質。最適化はコンパイラが勝手に判断する。

### `inline` で異なる定義を書いた場合

`inline` で **異なる定義** を複数の `.cpp` に置くと、これは ODR 違反だが：

> **コンパイルもリンクも通ってしまい、実行時に未定義動作**

```cpp
// a.cpp
inline int square(int x) { return x * x; }   // 25 を期待
// b.cpp
inline int square(int x) { return x + x; }   // 10 を期待
```

リンカは weak symbol を見つけたら **どちらか1つだけ採用** して残りを破棄する。中身は比較しない。
→ 自分が書いた定義が呼ばれない可能性、最適化レベルで結果が変わる、再現性のないバグの温床。

**安全に使うルール**：
- `inline` 関数／変数は **必ずヘッダに書く**
- マクロで条件分岐させない
- ヘッダ修正時はクリーンビルド

### クラス定義をヘッダに書く理由

ヘッダ／ソース分割は「めんどくさい慣習」ではなく、**コンパイル単位ごとの独立性と多重定義回避を両立する仕組み**。

| ファイル | 何を書く | 役割 |
|---|---|---|
| `.hpp`（ヘッダ） | クラスの形・関数のシグネチャ | **インターフェース** |
| `.cpp`（ソース） | 関数の本体 | **実装** |

#### なぜクラス定義は `.hpp` に書くのか

クラスを使う側は **そのクラスのサイズや構造を知らないとコンパイルできない**。

```cpp
// other.cpp
#include "myclass.hpp"   // ← これがないと…
MyClass obj;             // ❌ サイズが分からない、コンパイルエラー
```

クラス定義（`{ ... }` 部分）は **使う側のすべての `.cpp` に必要** → ヘッダに置く。

#### 関数本体までヘッダに書くと多重定義になる

```cpp
// myclass.hpp
class MyClass {
public:
    void foo() { /* 実装 */ }   // ← 本体をヘッダに書くと…
};
```

```cpp
// a.cpp と b.cpp が両方 #include すると
// → リンクエラー：multiple definition of `MyClass::foo'
```

#### ただしクラス定義内に書いた関数は自動 `inline`

```cpp
// myclass.hpp
class MyClass {
public:
    void foo() { /* 実装 */ }   // ✅ 自動的に inline、多重定義 OK
};
```

クラス定義の `{}` の中に書いた関数本体は **暗黙的に `inline` 扱い** になる。これが「短い関数はクラス内に書くと楽」の根拠。

### ODR の特例（多重定義が許される物）

| 種類 | 多重定義 OK？ | 仕組み |
|---|---|---|
| クラス定義 | ✅（全 `.cpp` で同一なら） | ODR の特例 |
| クラス内に書いた関数本体 | ✅ | 暗黙的に `inline` |
| クラス外の普通の関数本体 | ❌ | strong symbol |
| クラス外の関数本体（`inline` 付き） | ✅ | weak symbol |
| テンプレート | ✅ | 暗黙的に `inline` |
| `enum` | ✅（全 `.cpp` で同一なら） | ODR の特例 |

ヘッダに書く価値があるものは、ほとんど ODR の特例で許されている。

### ヘッダだけで関数を呼べる仕組み

ヘッダには宣言しかないのに、`#include "foo.hpp"` するだけで関数を呼べる。これは **コンパイル時とリンク時で必要な情報が違う** から。

| 段階 | 必要な情報 | どこから来るか |
|---|---|---|
| `obj.foo()` と書く | foo の引数・戻り値の型 | **ヘッダ**（宣言） |
| 呼び出しの機械語生成 | 「呼ぶ」コードを作る（中身は未定） | **ヘッダ** |
| 実行時の foo の処理 | foo の実体（機械語） | **`.o`**（実装） |

#### 流れ

```
main.cpp（ヘッダ #include）
   ↓ コンパイル
main.o：「foo を呼ぶ、でもアドレスは ???（未解決）」

myclass.cpp（実装）
   ↓ コンパイル
myclass.o：「foo はここにある！」

   ↓ リンカが結合
app（実行ファイル）：呼び出しのアドレスが確定
```

ヘッダで **型** が分かれば呼び出しコードは生成できる。**実体のアドレスは後回し**でよい、というのが分割コンパイルの本質。

### リンクし忘れるとどうなる

```bash
g++ main.cpp -o app           # myclass.cpp を入れ忘れた
```

→ コンパイルは通る、リンクで失敗：

```
undefined reference to `MyClass::foo()'
```

「**未解決の外部参照**」エラー。コンパイル時は宣言だけで通る、リンク時に実体がないと初めて気付く。

### 使い分けの目安

| 関数の性質 | どこに書くか |
|---|---|
| 1〜2行の短い関数（getter, setter） | クラス内（自動 inline） |
| 中〜重い処理 | `.cpp`（宣言だけヘッダに） |
| テンプレート関数 | ヘッダ（必須） |
| 共通の小さな関数 | ヘッダに `inline` 付き |

### 現代的な代替（C++17 以降）

```cpp
// ヘッダに直接書ける
inline int counter = 0;            // inline 変数
constexpr int MAX = 100;           // コンパイル時定数
inline constexpr double PI = 3.14; // 両方付けてもOK
```

`inline` 変数のおかげで、`extern` + 別 `.cpp` での定義という二段構えが省略できるようになった。

### ヘッダオンリーライブラリ

最近はあえて全部ヘッダに書くスタイルもある。

```cpp
// json.hpp
#pragma once
class Json {
public:
    void parse(...) { /* 実装 */ }
};
```

`inline` やテンプレートを駆使して、ヘッダ1つだけで使えるライブラリ。
代表例：`nlohmann/json`、`Catch2`、`spdlog`（一部）。

利点：`#include` するだけ、ビルド設定不要。
欠点：コンパイル時間が長くなる。

## C#との比較

| 項目 | C# | C++ |
|---|---|---|
| コンパイル単位 | プロジェクト全体 | `.cpp` 1つずつ |
| 中間生成物 | プロジェクト単位の `.dll` / `.exe` | ファイル単位の `.o` |
| 型情報の伝達 | コンパイラがプロジェクト全体を見渡す | `#include` で各 `.cpp` に手動配布 |
| 宣言と実装の分離 | 不要（同じファイル） | 必要（ヘッダ／ソース） |
| 多重定義の心配 | なし | あり（だから ODR がある） |
| `extern` 相当 | なし（クラスの static メンバが代替） | `extern int x;` |
| ヘッダの存在 | なし | 必須 |
| 増分ビルド | プロジェクト単位 | ファイル単位 |
| リンク | アセンブリ参照 | リンカでシンボル結合 |

C# は「**コンパイラが全部見えている**」前提。アセンブリ単位で一括コンパイルするので、矛盾した定義は即エラー、宣言と定義の分離も不要。

C++ は「**ファイルを独立にコンパイルする**」前提。性能（増分ビルド・並列ビルド）と引き換えに、ファイル間の整合性は **プログラマが保証** する必要がある。だからヘッダや `extern`、`inline` のような仕組みが要る。

```csharp
// C#（1ファイル完結）
public class MyClass {
    public void Foo() { /* 実装 */ }
}
var obj = new MyClass();
obj.Foo();
```

```cpp
// C++（ヘッダとソースに分割）
// myclass.hpp
class MyClass { public: void foo(); };

// myclass.cpp
void MyClass::foo() { /* 実装 */ }

// main.cpp
#include "myclass.hpp"
MyClass obj; obj.foo();
```

## 補足

### ヘッダに書ける物の整理

| 書く | 書かない（または `inline`） |
|---|---|
| クラス／構造体定義 | 関数の本体（`inline` なら OK） |
| `enum` | グローバル変数（`inline` なら OK） |
| テンプレート | 通常の関数本体 |
| `inline` 関数 | `static` でない `int x = 0;` 等 |
| `inline` 変数（C++17） | |
| `constexpr` 関数／変数 | |

### `static` と `inline` の違い

似たヘッダ用途で迷う2つ。

```cpp
// header.hpp
static int x = 0;        // 各 .cpp ごとに別実体
inline int y = 0;        // 全 .cpp で同じ実体
```

| 指定 | 各 `.cpp` の動き | 結果 |
|---|---|---|
| `static` | 独立した実体 | 別々の変数 |
| `inline` | weak symbol → 統合 | 同じ変数 |
| なし | strong symbol → 衝突 | リンクエラー |

### `const` 変数の特殊扱い

`const` 変数は **デフォルトで内部リンケージ**（その `.cpp` 内だけ有効）。共有したければ `extern`：

```cpp
// header.hpp
extern const int MAX;        // 宣言

// .cpp
const int MAX = 100;         // 定義（共有される）
```

C++17 以降は `inline constexpr` のほうが便利：

```cpp
// header.hpp に直接書ける
inline constexpr int MAX = 100;
```

### Windows と Linux の違い

| 役割 | Linux/MSYS2 | Windows MSVC |
|---|---|---|
| ソースファイル | `.cpp` | `.cpp` |
| オブジェクトファイル | `.o` | `.obj` |
| 静的ライブラリ | `.a` | `.lib` |
| 動的ライブラリ | `.so` | `.dll` |
| 実行ファイル | （拡張子なし） | `.exe` |

### ヘッダの命名

| 拡張子 | 意味 |
|---|---|
| `.h` | C 由来、C/C++ 両用 |
| `.hpp` | C++ 専用（一般的な慣習） |
| `.hxx` / `.hh` | あまり一般的でない |

標準ライブラリは拡張子なし（`<iostream>`, `<vector>`）。

### `inline` 違反の検出

通常のビルドでは **何も警告されない**。LTO（Link Time Optimization）を有効にすると検出されることがある：

```bash
g++ -flto a.cpp b.cpp -o app
# → warning: type ... violates the C++ One Definition Rule
```

ただし常に検出されるわけではない。`inline` の安全性はプログラマ責任。

### モジュール（C++20）

C++20 で導入された **モジュール** は、`#include` の問題を根本解決する仕組み。

```cpp
// foo.cppm
export module foo;
export class Foo { int x; };

// main.cpp
import foo;
```

C# の `using` に近い。ヘッダ／ソース分割の必要が大幅に減る。普及はまだ進行中。

### よくあるエラー

#### コンパイルエラー（`.cpp` → `.o`）

```
error: 'foo' was not declared in this scope
```

→ 宣言が見えていない。ヘッダの `#include` 漏れなど。

#### リンクエラー（`.o` → `.exe`）

```
undefined reference to `add(int, int)`
```

→ 関数の実体がリンク対象に含まれていない。`.cpp` をビルド対象から外していないか確認。

```
multiple definition of `foo'
```

→ 同じ関数が複数の `.o` で定義されている。ヘッダに `inline` なしで関数本体を書いたケースが典型。
