# 第37章: 演算子のムーブ対応

## 要点

### 普通の `operator+` の問題点

数値クラスを作って `+` を実装すると、典型的にはこう書く。

```cpp
class Numeric {
    std::vector<int> data;   // 大きなデータを持つ前提
public:
    Numeric& operator+=(const Numeric& other) { /* ... */ return *this; }
};

// 二項演算は += を使って実装するのが定石
Numeric operator+(const Numeric& a, const Numeric& b) {
    Numeric result = a;   // ① a をコピー
    result += b;          // ② b を足し込む
    return result;        // ③ 戻す（NRVO で最適化）
}
```

これは `a + b` 単独なら問題ないが、**`a + b + c` のような連続演算で無駄が出る**。

### 連続演算で何が起きるか

```cpp
Numeric x = a + b + c;
```

評価は左から：

```
ステップ1:  tmp1 = a + b      ← 一時オブジェクト tmp1 ができる
ステップ2:  x   = tmp1 + c    ← さらに tmp1 + c を計算
ステップ3:  tmp1 が破棄される
```

問題は2行目。**`tmp1` はもうすぐ捨てる予定** なのに、`operator+(const Numeric&, const Numeric&)` の中で：

```cpp
Numeric result = a;   // ← tmp1 をわざわざコピー
```

捨てる物をコピーしている。これがもったいない。

### 解決策：rvalue 用のオーバーロードを追加

`tmp1` のような「もうすぐ捨てる物」は **rvalue**。`T&&` で受ければ区別できる。

```cpp
// rvalue を受ける版
Numeric operator+(Numeric&& a, const Numeric& b) {
    a += b;                    // a は捨てる予定 → 直接書き換えてOK
    return std::move(a);       // 中身を奪って戻す（コピーしない）
}
```

ポイント：
- 引数 `a` は rvalue（`tmp1` など）→ **中身を奪ってよい**
- 新しいオブジェクトを作らず、`a` を直接更新
- 戻り値は `std::move(a)` でムーブ

### 4つのオーバーロードパターン

`+` の左右はそれぞれ lvalue / rvalue の2通り → 全部で4通り。

| 左 | 右 | 例 | 戦略 |
|---|---|---|---|
| lvalue | lvalue | `a + b` | 左をコピーして += |
| **rvalue** | lvalue | `(a+b) + c` | 左を奪う |
| lvalue | **rvalue** | `a + (b+c)` | 右を奪う（可換なら） |
| **rvalue** | **rvalue** | `(a+b) + (c+d)` | 左を奪う |

### 4オーバーロードの実装

```cpp
// ① lvalue + lvalue
Numeric operator+(const Numeric& a, const Numeric& b) {
    Numeric result = a;
    result += b;
    return result;
}

// ② rvalue + lvalue（左を奪う）
Numeric operator+(Numeric&& a, const Numeric& b) {
    a += b;
    return std::move(a);
}

// ③ lvalue + rvalue（右を奪う・可換のとき）
Numeric operator+(const Numeric& a, Numeric&& b) {
    b += a;
    return std::move(b);
}

// ④ rvalue + rvalue（左を奪う）
Numeric operator+(Numeric&& a, Numeric&& b) {
    a += b;
    return std::move(a);
}
```

### オーバーロード解決の動き

書くコードは変えなくていい。コンパイラが値カテゴリを見て自動で選ぶ。

```cpp
Numeric a, b, c, d;

a + b;            // 両方 lvalue → ①
(a+b) + c;        // 左 rvalue（一時物）、右 lvalue → ②
a + (b+c);        // 左 lvalue、右 rvalue → ③
(a+b) + (c+d);    // 両方 rvalue → ④
```

### `std::move(a)` を忘れないこと

```cpp
Numeric operator+(Numeric&& a, const Numeric& b) {
    a += b;
    return a;             // ❌ a は名前付きなので lvalue 扱い → コピー
    return std::move(a);  // ✅ ムーブで返る
}
```

第34章の **「名前が付いたら lvalue」ルール**。引数 `a` の型は `Numeric&&` でも、変数名として書いた瞬間 lvalue になる。明示的に `std::move` で rvalue 化しないとコピーが走る。

### NRVO と `std::move` の使い分け

```cpp
// ローカル変数を返す → NRVO に任せる
Numeric operator+(const Numeric& a, const Numeric& b) {
    Numeric result = a;
    result += b;
    return result;             // ✅ NRVO（書かない方が良い）
    // return std::move(result); ← 書くと最適化を阻害する
}

// 引数を返す → NRVO の対象外、std::move 必須
Numeric operator+(Numeric&& a, const Numeric& b) {
    a += b;
    return std::move(a);       // ✅ 必要
}
```

ルール：
- **ローカル変数の `return` は `std::move` を書かない**（NRVO に任せる）
- **関数引数の `return` は `std::move` を書く**（NRVO の対象外）

### 効果のイメージ

`a + b + c + d`（大きな vector を持つ Numeric）の場合：

| 実装 | コピー回数 |
|---|---|
| `const T&` だけ | 3回（毎回 result を作成） |
| rvalue オーバーロード追加 | **1回**（最初の `a + b` のみ） |

連続演算が多い数値計算では効果が大きい。

## C#との比較

| 項目 | C# | C++ |
|---|---|---|
| 演算子オーバーロード | 可能（`static` メソッドとして） | 可能（メンバ／非メンバ関数） |
| 値カテゴリ別のオーバーロード | **できない** | `T&` / `T&&` で区別 |
| 連続演算での一時物の最適化 | 言語的に手段なし（GC任せ） | rvalue オーバーロードで明示的最適化 |
| 一時オブジェクトの扱い | 区別しない（参照型は全部参照） | lvalue / rvalue で型レベルに区別 |
| 戻り値最適化 | 値型は構造体で多少効くが意識する場面なし | NRVO / RVO が言語仕様 |

C# では「a + b + c」と書いても、参照型ならポインタの差し替えだけ（中身は1つ）。
C++ では値セマンティクスのため、各演算で新しいオブジェクトが生まれる → だからこそ rvalue 版で **使い捨ての一時物を再利用** する最適化が意味を持つ。

## 補足

### なぜ `+=` を先に実装するのか

二項演算の定石は「`+=` を実装してから `+` を `+=` で組み立てる」。

```cpp
class Numeric {
public:
    Numeric& operator+=(const Numeric& other);   // ← まずこれ
};

// + は += を使って実装
Numeric operator+(const Numeric& a, const Numeric& b) {
    Numeric result = a;
    result += b;
    return result;
}
```

理由：
- `+=` は1関数で済む
- `+` は `+=` の上に薄く乗るだけ
- ロジックの重複がなくなる

### 可換でない演算（`-` など）

`+` は可換（`a + b == b + a`）なので「右を奪う」最適化ができたが、`-` は不可換。

```cpp
// ② rvalue - lvalue（左を奪える）
Numeric operator-(Numeric&& a, const Numeric& b) {
    a -= b;
    return std::move(a);
}

// ③ lvalue - rvalue（右を奪うと b - a になってしまう）
Numeric operator-(const Numeric& a, Numeric&& b) {
    Numeric result = a;       // 普通にコピー
    result -= b;
    return result;
    // または b の中身を一度退避して計算する手法もあるが複雑
}
```

非可換演算子では rvalue 最適化が部分的にしか効かない。

### ④を ②③ で代用する省略形

`rvalue + rvalue`（パターン④）は、片方を `const T&` 経由で②に丸投げできる。

```cpp
Numeric operator+(Numeric&& a, Numeric&& b) {
    return std::move(a) + b;   // ② に流す
}
```

ただし明示的に書いた方が読みやすいので、4つ全部書くスタイルも一般的。

### 標準ライブラリでの実例

`std::string` も同じパターンを実装している。

```cpp
std::string s = "a";
auto r = s + "b" + "c" + "d";   // 連続結合
// 内部で rvalue オーバーロードが選ばれ、不要なコピーが省略される
```

`std::string::operator+` には rvalue 版が複数定義されており、一時 string を使い回す最適化が効く。

### `*=` や `/=` も同じパターン

`*`, `/`, `%` などの二項演算子はすべて同じ4オーバーロードパターンが適用できる。

```cpp
Numeric operator*(const Numeric& a, const Numeric& b);
Numeric operator*(Numeric&& a, const Numeric& b);
Numeric operator*(const Numeric& a, Numeric&& b);
Numeric operator*(Numeric&& a, Numeric&& b);
```

ボイラープレートは多いが、書く価値があるのは「連続演算が頻出する重い型」だけ。

### 軽い型（`int` ラッパー等）には不要

```cpp
class SmallInt {
    int value;   // 中身が小さい → コピーもムーブも同じコスト
};
```

中身がスタック上の小さな値だけなら、`const T&` 1個だけで十分。ムーブしてもコピーと同じなので、4オーバーロードは無駄に複雑なだけ。

**ヒープリソースを持つ型でだけ意味がある最適化**。

### Rule of Zero との関係

メンバが `std::vector` などの標準型だけなら、ムーブコンストラクタやムーブ代入は自動生成される（Rule of Zero）。
**演算子のムーブ対応は別の話** で、こちらは自分で書く必要がある。

```cpp
class Numeric {
    std::vector<int> data;   // ムーブは自動生成される
};

// ↓ 演算子のムーブ対応は自前で書く必要あり
Numeric operator+(Numeric&& a, const Numeric& b) { ... }
```

### 戻り値の `Numeric&&` ではなく `Numeric` で返す理由

```cpp
// ❌ 危険：参照を返す
Numeric&& operator+(Numeric&& a, const Numeric& b) {
    a += b;
    return std::move(a);   // a は呼び出し側の一時オブジェクトかもしれない
}

// ✅ 値で返す
Numeric operator+(Numeric&& a, const Numeric& b) {
    a += b;
    return std::move(a);
}
```

`Numeric&&` で返すと、引数として渡された一時オブジェクトへの参照を返すことになり、寿命切れの危険がある。値で返せばムーブされて新しいオブジェクトになるので安全。

### この章のキーフレーズまとめ

- **「捨てる物はコピーするな、奪え」**
- 連続演算で生まれる一時オブジェクトを使い回す
- 区別する手段は `T&&`
- 4つのオーバーロード（lvalue/rvalue × 左/右）
- 引数を返すときは `std::move` を忘れない（ローカル変数では不要）

## 理解度

- [x] 普通の `operator+` が連続演算で無駄なコピーを生む理由
- [x] 一時オブジェクト（rvalue）を奪うための `T&&` オーバーロード
- [x] 4つのオーバーロードパターン（lvalue/rvalue × 2）
- [x] 可換演算（`+`）と非可換演算（`-`）での扱いの違い
- [x] `std::move(a)` が必要な理由（引数は名前付き → lvalue 扱い）
- [x] NRVO が効く場面と効かない場面の使い分け
- [x] `+=` を先に実装してから `+` を組み立てる定石
- [x] 戻り値は `Numeric` で返す（`Numeric&&` は危険）
- [x] 軽い型では4オーバーロード不要、ヒープリソースを持つ型でのみ意味がある
- [x] Rule of Zero でクラスのムーブは自動、演算子のムーブ対応は自前
