# 第4章: エラーと警告

## 要点

### 文法エラー（syntax error）

- コードが C++ の文法に従っていない場合に発生
- コンパイラが解析段階で検出する

```cpp
int x = ;       // セミコロンの前に値がない
int 123abc = 0; // 変数名が数字で始まっている
```

### 意味エラー（semantic error）

- 文法は正しいが、意味的に正しくないコード
- 型の不一致や未定義の変数の使用など

```cpp
int x = "hello";  // int に文字列リテラルは代入できない
std::cout << y;    // y が宣言されていない
```

### コンパイラオプションの重要性

- `-Wall` / `-pedantic-errors` を**つけないと**、問題のあるコードが警告もエラーも出さずに通ってしまう
- 常にこれらのオプションを有効にして開発することが重要

```bash
# 良い例
g++ -std=c++17 -Wall -pedantic-errors main.cpp

# 悪い例（警告が出ない）
g++ main.cpp
```

## C#との比較

| 項目 | C# | C++ |
|------|-----|------|
| エラー検出タイミング | コンパイル時 + IDE がリアルタイムで表示 | コンパイル時（`g++` 実行時） |
| 警告の厳格さ | デフォルトで比較的厳格 | **オプションなしだと甘い** → `-Wall` 必須 |
| エラーメッセージ | 比較的わかりやすい | テンプレート絡みだと非常に長くなることがある |

- C# は Visual Studio / Rider が常時エラーを表示してくれるが、C++ のコマンドライン開発では `g++` を実行するまでエラーがわからない
- C# の `#nullable enable` のように、C++ も `-Wall` で「厳格モード」を有効にするイメージ

## 補足

- エラーメッセージは**上から順に**読む。最初のエラーが後続のエラーの原因になっていることが多い
- テンプレート関連のエラーは長大になりがち。最初の数行に注目するのがコツ
- `-Wextra` をさらに追加するとより多くの警告を検出できる
