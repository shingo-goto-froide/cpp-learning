# 第29章: 動的メモリー確保

## 要点

### なぜ動的メモリー確保が必要か

- スタック上の変数は**スコープを抜けると消える**
- 実行時まで必要なサイズが分からないデータはスタックに置けない
- **ヒープ**に確保し、必要な間だけ保持 → 動的メモリー確保

### 3つのレベルを区別する

```
new 式          ← ユーザーが使う構文（確保 + コンストラクタ）
  ├─ operator new   ← 生メモリー確保関数（C++流）
  └─ コンストラクタ呼出

malloc          ← C由来の生メモリー確保関数
```

### `malloc` / `free`（C由来）

```cpp
void* malloc(std::size_t size);
void  free(void* ptr);
```

- **生のメモリー（型のないバイト列）**を確保／解放するだけ
- 戻り値は `void*`（型未定の汎用ポインター）
- 失敗時は `nullptr` を返す
- `free(nullptr)` は安全（何もしない、と規格で保証）

### `malloc` を使う3ステップ

```cpp
// ① 確保：生のメモリーを size バイト分もらう
void* raw = std::malloc(sizeof(int));

// ② 型変換：void* を int* に変換
int* p = static_cast<int*>(raw);

// ③ 代入：確保した領域に値を書き込む
*p = 42;

std::free(p);   // 解放
```

- ②と③の間は**未初期化**（読むと未定義動作）
- 1行にまとめることも多い

```cpp
int* p = static_cast<int*>(std::malloc(sizeof(int)));
```

### `operator new` / `operator delete`（C++）

```cpp
void* operator new(std::size_t size);
void  operator delete(void* ptr) noexcept;
```

- シグネチャは `malloc`/`free` と類似（引数・戻り値の型が同じ）
- 違い：
  - 確保失敗時に **`std::bad_alloc` を投げる**
  - **クラスごとにオーバーロード可能**
  - **グローバル置換可能**（全 `new` を差し替えられる）
- 生メモリー確保だけ。コンストラクタは呼ばない。

### `new` 式との関係（2段階）

```cpp
// new T(args) と書くと実質：
void* mem = operator new(sizeof(T));  // ① メモリー確保
T* p = new(mem) T(args);               // ② コンストラクタ呼出（placement new）

// delete p も2段階：
p->~T();                // ① デストラクタ呼出
operator delete(p);     // ② メモリー解放
```

### `malloc` と `new` の使い分け

| | `malloc` | `new` |
|---|---|---|
| メモリー確保 | ○ | ○ |
| コンストラクタ呼出 | ✗ | ○ |
| デストラクタ呼出（解放時） | ✗ | ○（`delete`） |
| 型情報 | なし（`void*`） | あり（`T*`） |
| 失敗時 | `nullptr` | 例外 |

- **クラス型**（`vector`, `string`, 自作クラス等）→ 必ず `new`/`delete`
- **組込型** → どちらでも動くが `new` 推奨
- **生のバイト列が欲しい** → `malloc`

### `vector` を `malloc` で確保してはいけない

```cpp
// 危険！絶対に書いてはいけない
auto v = static_cast<std::vector<int>*>(
    std::malloc(sizeof(std::vector<int>))
);
v->push_back(1);   // 未定義動作（内部ポインターがゴミ値）
```

- `vector` の内部状態（ポインター、サイズ、容量）は**コンストラクタで初期化**される
- `malloc` はバイト列を返すだけなので、内部は未初期化のまま

### スマートポインター（RAII による自動解放）

```cpp
#include <memory>

void func() {
    auto p = std::make_unique<int>(42);
    *p = 100;
}   // ← スコープ終了で自動 delete
```

- デストラクタで `delete` するクラス
- **スコープを抜けると必ず解放される**（例外時も）

| 種類 | 所有権 | 用途 |
|---|---|---|
| `std::unique_ptr` | 単独（ムーブのみ） | 最も基本 |
| `std::shared_ptr` | 共有（参照カウント） | 複数箇所で保持 |
| `std::weak_ptr` | 非所有 | 循環参照の回避 |

### モダンC++の推奨

```cpp
// ❌ 古い書き方
int* p = new int(42);
delete p;

// ✅ モダンC++
auto p = std::make_unique<int>(42);
```

- 生の `new`/`delete` はほぼ書かないのが作法

## C#との比較

| 項目 | C# | C++ |
|---|---|---|
| メモリー確保 | `new`（GC前提） | `malloc` / `new` / `operator new` |
| 解放 | GCが自動 | `free` / `delete` で明示 |
| 初期化と確保 | 常に一体 | **分離可能**（placement newなど） |
| `object` に相当 | `object`（全型の基底） | `void*`（全ポインターの汎用型） |
| 確保失敗 | `OutOfMemoryException` | `std::bad_alloc`（`new`） |
| リーク | 基本発生しにくい | 明示解放忘れで即発生 |
| ダングリング参照 | 発生しにくい | 頻出する落とし穴 |
| 自動解放の仕組み | GC（非決定的） | RAII + スマートポインター（**決定的**） |
| `using` / `IDisposable` | 明示的に書く必要 | スコープ宣言だけで発動 |

## 補足

### シグネチャの正確な意味

C++規格では関数のシグネチャは以下を含む：

- 関数名
- 引数の型リスト
- （テンプレートなら）テンプレート引数
- （メンバーなら）const/volatile/参照修飾

**戻り値の型は含まない**（テンプレートを除く）。

`malloc` と `operator new` は「引数・戻り値の型が似ている」だけで、名前が違うのでシグネチャは**別物**。

### 生のメモリー（raw memory）とは

「**型情報も初期値もないバイト列**」のこと。

```
生のメモリー（バイト列）
       ↓ コンストラクタ
構築済みオブジェクト（意味のある値）
       ↓ デストラクタ
生のメモリーに戻る
       ↓ 解放
OSに返却
```

`malloc` / `operator new` は①の段階まで。`new` 式は②まで自動でやってくれる。

### 確保と初期化を分離する場面

通常のアプリ開発ではほぼ不要。登場するのは：

1. **`std::vector` などコンテナの実装**（capacity 分だけ先に確保し、要素は後で構築）
2. **メモリープール / アロケーター**（高頻度確保を高速化）
3. **オブジェクトの再構築**（同じ領域を使い回す）
4. **`std::optional` / `std::variant`**（値を持つ／持たないの切替）
5. **共有メモリー・ファイルマップ**（既存領域に構築）
6. **例外安全**（確保失敗でコンストラクタを呼ばずに済む）

### `void*` の扱い

- どんな型のポインターも `void*` に**暗黙変換**される（キャスト不要）
- 逆は**明示キャストが必要**（`static_cast<T*>`）
- `void*` のままでは**デリファレンスできない**

### メモリーリークとダングリング

```cpp
// メモリーリーク：解放忘れ
void leak() {
    std::malloc(1000);   // 誰もアドレスを持たない → 解放不能
}

// ダングリングポインター：解放後に使う
std::free(p);
*p = 10;        // 危険！
std::free(p);   // 二重解放も危険！
```

- `free`/`delete` した後のポインターは使ってはいけない
- 二重解放も未定義動作

### 例外安全の観点

```cpp
void func() {
    int* p = new int(42);
    might_throw();   // 例外発生！
    delete p;        // ← 実行されない → リーク
}
```

スマートポインターなら例外時もデストラクタが走るので安全：

```cpp
void func() {
    auto p = std::make_unique<int>(42);
    might_throw();   // 例外でも p のデストラクタは呼ばれる
}
```

### RAII（Resource Acquisition Is Initialization）

- 「リソースの獲得は初期化時に、解放はデストラクタで」という設計原則
- C++は**スコープを抜ける時に必ずデストラクタを呼ぶ**ため成立
- メモリーだけでなく、ファイル・ロック・接続など全リソースに適用可能
- C# の `using` より強力（型に組み込めば常に発動）

## 理解度

- [x] 動的メモリー確保が必要な理由（スタックの寿命・サイズ制約）
- [x] `malloc`/`free` の役割と `void*` を返す理由
- [x] `operator new`/`operator delete` と `malloc`/`free` の違い
- [x] `new` 式が「確保 + コンストラクタ」の2段階である仕組み
- [x] `malloc` の使用3ステップ（確保 → 型変換 → 代入）
- [x] クラス型を `malloc` で確保してはいけない理由
- [x] シグネチャの正確な定義（関数名を含む）
- [x] 「生のメモリー」と「構築済みオブジェクト」の区別
- [x] 確保と初期化を分離する具体的場面
- [x] スマートポインターによる RAII 自動解放
- [x] `unique_ptr` / `shared_ptr` / `weak_ptr` の使い分け
- [x] モダンC++では生 `new`/`delete` を避ける作法
