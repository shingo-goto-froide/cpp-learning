# 第35章: ムーブの実装

## 要点

### `&` と `&&` の違い

C++のリファレンスには2種類ある。どちらも「別名」であってポインタではない。

| 記号 | 名前 | 束縛できるもの |
|---|---|---|
| `T&` | lvalue リファレンス | lvalue（名前のあるオブジェクト） |
| `T&&` | rvalue リファレンス | rvalue（一時オブジェクトや `std::move` した物） |
| `const T&` | const lvalue リファレンス | lvalue と rvalue の両方（特例） |

```cpp
int x = 10;

int&  lr  = x;             // ✅ T& は lvalue を受ける
int&  lr2 = 42;            // ❌ rvalue は受けられない

int&& rr  = 42;            // ✅ T&& は rvalue を受ける
int&& rr2 = std::move(x);  // ✅ std::move で rvalue 化した物
int&& rr3 = x;             // ❌ lvalue は受けられない

const int& cr = 42;        // ✅ const T& は両方OK
```

### なぜ `&&` が必要か：オーバーロードでムーブとコピーを切り替える

```cpp
void f(const std::string& s);  // lvalue 用 → コピー処理
void f(std::string&& s);       // rvalue 用 → ムーブ処理

std::string a = "hello";
f(a);              // lvalue 版（コピー）
f(std::move(a));   // rvalue 版（ムーブ）
f("world");        // 一時オブジェクト → rvalue 版（ムーブ）
```

`&&` がなければ「奪っていい一時物」と「使い続ける lvalue」を型で区別できない。

### Rule of Five（5原則）

リソース管理が必要なクラスで、コンパイラが自動生成するデフォルト挙動が正しくない場合、以下の5つの特殊メンバ関数を **1セットで定義する**。

| # | 名前 | シグネチャ |
|---|---|---|
| 1 | デストラクタ | `~T()` |
| 2 | コピーコンストラクタ | `T(const T&)` |
| 3 | コピー代入演算子 | `T& operator=(const T&)` |
| 4 | ムーブコンストラクタ | `T(T&&) noexcept` |
| 5 | ムーブ代入演算子 | `T& operator=(T&&) noexcept` |

C++11 以前の **Rule of Three**（1〜3）にムーブが加わって5つになった。

### Rule of Five の完全実装

```cpp
class MyArray {
    int* data;
    size_t size;
public:
    MyArray(size_t n) : data(new int[n]), size(n) {}

    // 1. デストラクタ
    ~MyArray() { delete[] data; }

    // 2. コピーコンストラクタ（ディープコピー）
    MyArray(const MyArray& other)
        : data(new int[other.size]), size(other.size) {
        std::copy(other.data, other.data + size, data);
    }

    // 3. コピー代入演算子
    MyArray& operator=(const MyArray& other) {
        if (this != &other) {
            int* new_data = new int[other.size];
            std::copy(other.data, other.data + other.size, new_data);
            delete[] data;
            data = new_data;
            size = other.size;
        }
        return *this;
    }

    // 4. ムーブコンストラクタ
    MyArray(MyArray&& other) noexcept
        : data(other.data), size(other.size) {
        other.data = nullptr;   // ← 元を空にする（重要）
        other.size = 0;
    }

    // 5. ムーブ代入演算子
    MyArray& operator=(MyArray&& other) noexcept {
        if (this != &other) {
            delete[] data;           // 既存リソースを解放
            data = other.data;       // ポインタを奪う
            size = other.size;
            other.data = nullptr;    // 元を空に
            other.size = 0;
        }
        return *this;
    }
};
```

### ムーブ実装の2ステップ

ムーブコンストラクタ／代入演算子の本体は2つだけ：

1. **ポインタ（リソース）を奪う**
2. **元を `nullptr`（または空状態）にする**

```cpp
MyArray(MyArray&& other) noexcept {
    data = other.data;       // ① 奪う
    other.data = nullptr;    // ② 元を空に（忘れると二重解放）
}
```

`other.data = nullptr` を忘れると、両方が同じヒープ領域を指したままになり、デストラクタで二重解放してクラッシュする。

### ムーブ代入では既存リソースの解放が必要

```cpp
MyArray& operator=(MyArray&& other) noexcept {
    if (this != &other) {
        delete[] data;           // ← コピーと違って忘れがち
        data = other.data;
        other.data = nullptr;
    }
    return *this;
}
```

代入なので `*this` は既に何かを所有している → 先に解放しないとリーク。

### `noexcept` を必ず付ける

ムーブコンストラクタ／代入演算子には `noexcept` を付ける。

```cpp
MyArray(MyArray&& other) noexcept { ... }
```

理由：`std::vector` などの標準コンテナは、要素のムーブが `noexcept` でないと **コピーにフォールバック** する（例外安全のため）。`noexcept` がないとムーブが効かなくなる。

### 自動生成のルール

| ユーザーが書いたもの | 自動生成されるもの |
|---|---|
| なし | コピー・ムーブ・デストラクタ全部 |
| デストラクタ | コピーは生成、**ムーブは生成されない** |
| コピー（どれか） | **ムーブは生成されない** |
| ムーブ（どれか） | **コピーは `delete` 扱い** |

ムーブを書いた瞬間にコピーが消えるなど、意図せず制限が変わる。これも「全部書け」の理由のひとつ。

### 「名前が付いたら lvalue」ルールの落とし穴

ムーブコンストラクタの中でも、引数 `other` は **名前を持つので lvalue** として扱われる。

```cpp
void func(MyArray&& other) {       // other の型は MyArray&&
    MyArray a = other;             // ❌ コピーされる
                                   //（other は名前あり → lvalue）
    MyArray b = std::move(other);  // ✅ ムーブ
}
```

ムーブコンストラクタ／代入演算子の中で `other` のメンバを別オブジェクトに渡すときは、再度 `std::move` が必要。

### `= default` と `= delete` の活用

明示的にコピーやムーブの可否を指定できる。

```cpp
// コピー可能、ムーブ禁止
class CopyOnly {
    CopyOnly(const CopyOnly&) = default;
    CopyOnly(CopyOnly&&) = delete;
};

// コピー禁止、ムーブのみ可能
class MoveOnly {
    MoveOnly(const MoveOnly&) = delete;
    MoveOnly(MoveOnly&&) noexcept = default;
};
// std::unique_ptr / std::thread / std::mutex はこのパターン
```

### Rule of Zero（現代の推奨）

そもそも自分でリソース管理しないのが最善。生ポインタを持たず、`std::vector` / `std::string` / `std::unique_ptr` / `std::shared_ptr` で組み立てれば、Rule of Five は **何も書かなくてよい**。

```cpp
// ❌ 古典的（5関数全部書く）
class Buffer {
    int* data;
    size_t size;
    // デストラクタ、コピー2つ、ムーブ2つ全部必要
};

// ✅ 現代的（Rule of Zero）
class Buffer {
    std::vector<int> data;   // 全部 vector が面倒見てくれる
    std::string name;
};
```

### Rule of Five と Rule of Zero の使い分け

| 状況 | 適用するルール |
|---|---|
| 標準ライブラリだけで組み立てられる | **Rule of Zero** |
| 生ポインタ／OSリソース／Cライブラリハンドルを直接持つ | **Rule of Five** |
| 一部だけカスタムしたい | **Rule of Five を全部明示**（不要なものは `= default`） |

## C#との比較

| 項目 | C# | C++ |
|---|---|---|
| `&` 相当 | `ref` / `out`（参照渡し） | `T&` / `const T&` |
| `&&` 相当 | 存在しない | `T&&`（ムーブ用） |
| Rule of Five | 概念なし（GC管理） | 必要 |
| 確定解放 | `IDisposable` + `Dispose` パターン | デストラクタ |
| コピーの制御 | `ICloneable` / 自前 `Clone()` | コピーコンストラクタ |
| ムーブ | 存在しない | ムーブコンストラクタ／代入 |
| 推奨スタイル | `IDisposable` 実装 | **Rule of Zero**（標準型に任せる） |
| 二重解放 | 構造的に起こらない | 注意しないと起こる |
| `noexcept` 相当 | なし | ムーブには必須 |

C# の `Dispose` パターンは Rule of Five のうち「デストラクタ部分」だけを切り出したような物。コピー・ムーブの細かい制御は C++ 固有のテーマ。

## 補足

### なぜ「全部か、ゼロか」なのか

Rule of Five の本質は「**デフォルト挙動を1つでも上書きするなら、関連する挙動も全部見直せ**」。

```cpp
class MyArray {
    int* data;
public:
    MyArray(size_t n) : data(new int[n]) {}
    ~MyArray() { delete[] data; }   // ← これだけ書くと…
    // コピーコンストラクタは自動生成（浅いコピー）
};

MyArray a(10);
MyArray b = a;   // 浅いコピー → 二重解放でクラッシュ
```

デストラクタを書いた時点で、コピーやムーブもセットで考えないと壊れる可能性が高い。

### swap idiom（コピー＆スワップ）

代入演算子を1つの実装にまとめるテクニック。

```cpp
class MyArray {
    int* data;
    size_t size;

    void swap(MyArray& other) noexcept {
        std::swap(data, other.data);
        std::swap(size, other.size);
    }
public:
    // コピー＆ムーブ代入をまとめて記述
    MyArray& operator=(MyArray other) noexcept {  // 値渡し
        swap(other);
        return *this;
    }
};
```

- 引数を値渡しにする → 呼び出し側が lvalue ならコピー、rvalue ならムーブが自動選択
- 関数内では受け取った `other` を `swap` で交換
- 例外安全性も自然に確保される

ただし最近は明示的に2つ書くスタイルも多い（性能上の理由）。

### `noexcept` がないとどうなるか実例

```cpp
std::vector<MyClass> v;
v.reserve(100);
// ... v に要素追加 ...
v.push_back(MyClass{});   // 容量を超えると再確保

// 再確保時、既存要素を新しいメモリに移す必要がある
//  - MyClass のムーブが noexcept → ムーブで移す（速い）
//  - noexcept でない          → コピーで移す（遅い、ムーブの意味なし）
```

`noexcept` の有無で性能が大きく変わる。

### ムーブ後の状態：valid but unspecified

```cpp
std::string s1 = "hello";
std::string s2 = std::move(s1);
// s1 は「有効だが未規定の状態」
```

- ✅ できる：再代入、デストラクタ、`size()` / `empty()` / `clear()`
- ❌ してはいけない：`s1[0]`、`s1.front()` などの中身参照

実務の指針：**ムーブ後は再代入かスコープを抜けて捨てるまで触らない**。

### 自己代入のチェック

```cpp
MyArray& operator=(MyArray&& other) noexcept {
    if (this != &other) {     // ← 自己代入チェック
        delete[] data;
        ...
    }
    return *this;
}
```

`a = std::move(a);` のような自己ムーブで `delete[] data;` した後に `data = other.data;`（同じポインタ）してしまう事故を防ぐ。

### コピー代入での例外安全（強い保証）

```cpp
MyArray& operator=(const MyArray& other) {
    if (this != &other) {
        int* new_data = new int[other.size];   // ① 先に確保
        std::copy(...);                         // ② コピー
        delete[] data;                          // ③ 最後に旧領域解放
        data = new_data;
        size = other.size;
    }
    return *this;
}
```

`new` は失敗すると例外を投げる。**先に新しい領域を確保してから旧領域を解放する**ことで、例外時にも `*this` の状態が壊れない（強い例外保証）。

### Rule of Five の落とし穴：継承

基底クラスで `virtual` デストラクタを書くと、Rule of Five の対象になる。

```cpp
class Base {
public:
    virtual ~Base() = default;   // ← これを書いた瞬間…
    // コピー・ムーブは自動生成されなくなる場合がある
};
```

派生クラスを作る基底クラスでは、`= default` で5つ明示するのが安全。

### `unique_ptr` メンバを持つクラスは自動的に MoveOnly

```cpp
class Resource {
    std::unique_ptr<int> p;   // unique_ptr はコピー禁止
    // → このクラスもコピー禁止になる（自動生成のコピーが delete 扱い）
    // → ムーブのみ可能
};
```

`unique_ptr` を1つメンバに入れるだけで、何も書かずに「コピー禁止＋ムーブ可能」が手に入る。

### 5つを書く順番のおすすめ

1. デストラクタ（リソース解放を最初に）
2. ムーブコンストラクタ＋ムーブ代入（ポインタ奪取）
3. コピーコンストラクタ＋コピー代入（ディープコピー）

ムーブはコピーより簡単（メンバの付け替えだけ）なので先に書く方が楽。
