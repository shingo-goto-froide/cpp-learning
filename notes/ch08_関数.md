# 第8章: 関数

## 要点

### 関数の宣言と定義

- C++では関数を**使う前に宣言が必要**
- **宣言（declaration）**: 関数のシグネチャだけ。`;`で終わる
- **定義（definition）**: 実際の処理本体を含む

```cpp
// 宣言（プロトタイプ）
void f();

int main() {
    f(); // OK: 宣言済み
}

// 定義
void f() {
    std::cout << "hello"s;
}
```

- 宣言なしで呼ぶとコンパイルエラーになる

```cpp
int main() {
    f(); // エラー: fはまだ宣言されていない
}
void f() { /* ... */ }
```

- 実務ではヘッダーファイル（`.h`/`.hpp`）に宣言、ソースファイル（`.cpp`）に定義を書くのが一般的
- C++20のモジュール（`import`/`export`）でヘッダー不要になる流れだが、まだ発展途上

### 書籍がラムダを多用する理由

- ラムダは変数に代入する形なので、宣言と定義が分離しない
- 関数の宣言の煩わしさを避けて本題に集中するための書き方

```cpp
int main() {
    auto add = [](int a, int b) { return a + b; };
    add(1, 2); // OK: 宣言不要
}
```

- 実務ではラムダと通常の関数を使い分ける

| | ラムダ | 通常の関数 |
|---|---|---|
| 再利用 | その場限り向き | 複数ファイルから呼べる |
| 可読性 | 短い処理向き | 長い処理向き |
| 再帰 | やりにくい | 自然に書ける |

### エスケープシーケンス

- C#と同じエスケープシーケンスが使える

```cpp
std::cout << "4\t8"s;  // タブ区切りで出力: 4    8
```

| 記法 | 意味 |
|---|---|
| `\t` | タブ |
| `\n` | 改行 |
| `\\` | バックスラッシュ |
| `\"` | ダブルクォート |

### スタックとメモリ管理

- 関数のローカル変数はスタック領域に確保される
- スコープを抜けると自動的に解放される

```cpp
void f() {
    int x = 10;              // スタックに確保
    std::string s = "hello"; // オブジェクト自体はスタック、文字データはヒープ
}  // ← x, s ともに自動で解放される
```

- `std::string`のメモリ構造:

```
スタック                  ヒープ
┌──────────────┐        ┌───────────┐
│ s (管理情報)  │───────→│ "hello"   │
│ ・ポインタ    │        └───────────┘
│ ・サイズ      │
│ ・容量        │
└──────────────┘
```

### RAII（Resource Acquisition Is Initialization）

- 「コンストラクタで取得、デストラクタで解放」という原則
- スコープを抜けると**必ず**デストラクタが呼ばれ、リソースが解放される
- メモリだけでなくファイル・ソケット・ロックなどあらゆるリソースに適用

```cpp
{
    std::ifstream fs("a.txt");  // コンストラクタでファイルを開く
    // ...
}  // デストラクタでファイルを閉じる（自動）
```

### `new`/`delete`とスマートポインタ

- `new`で確保するとスコープに紐づかず、自動解放されない

```cpp
{
    std::string s = "hello";                   // スコープ抜けたら解放
    std::string* p = new std::string("hello"); // スコープ抜けても残る → リーク
}
```

- 現代C++では`new`/`delete`を直接書かず、**スマートポインタ**を使う

```cpp
#include <memory>

void f() {
    auto p = std::make_unique<std::string>("hello"); // ヒープに確保
}  // ← スマートポインタのデストラクタが自動でdelete（RAII）
```

| 方法 | 解放 | 使う場面 |
|---|---|---|
| スタック変数 | 自動 | 基本はこれ |
| `new`/`delete` | 手動 | 現代C++では避ける |
| スマートポインタ | 自動 | ヒープが必要な時はこれ |

### 末尾再帰の最適化（Tail Call Optimization）

- 再帰呼び出しが関数の最後の処理である場合、コンパイラがループに変換する最適化
- スタックの消費をなくし、スタックオーバーフローを防げる

```cpp
// 通常の再帰（末尾再帰でない）
int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);  // 戻り値に「n *」の計算が残っている
}

// 末尾再帰にしたもの
int factorial(int n, int acc = 1) {
    if (n <= 1) return acc;
    return factorial(n - 1, n * acc);  // これが最後の処理
}
```

- C++では言語仕様として**保証されていない**（コンパイラの最適化オプション次第）
- GCC/Clangは`-O2`以上で対応
- 確実にスタックオーバーフローを防ぎたいならループで書くのが安全

## C#との比較

| 項目 | C# | C++ |
|------|-----|------|
| 関数の宣言 | 不要（クラス内の定義順は自由） | 使う前に宣言が必要 |
| 宣言と定義の分離 | なし（メソッド定義のみ） | ヘッダーに宣言、ソースに定義 |
| メモリ管理 | GCが自動回収 | RAII（スコープ抜けで即解放） |
| ヒープのオブジェクト | GCが回収 | `delete`が必要（スマートポインタで自動化） |
| リソース解放 | `using`/`Dispose()`パターン | RAII（全リソース統一的に管理） |
| `using`の付け忘れ | GC頼みになる | そもそも起きない（デストラクタは必ず呼ばれる） |
| 末尾再帰の最適化 | 保証なし | 保証なし（どちらも同じ事情） |

## 補足

### C#のGCとC++のRAIIの根本的な違い

- C#: **メモリはGC、それ以外は`Dispose`パターン**（二重管理）
- C++: **全リソースをRAIIで統一管理**

```csharp
// C# — usingを書かないとDispose()されない
var fs = new FileStream("a.txt", FileMode.Open);
// ← usingがないとGC任せ（ファイルハンドルは閉じられない可能性）
```

```cpp
// C++ — 何も書かなくてもスコープ終了で必ずデストラクタが走る
{
    std::ifstream fs("a.txt");
}  // ← 必ず閉じられる。特別な構文は不要
```

### SSO（Small String Optimization）

- 短い文字列はヒープを使わずスタック内に収める最適化
- `std::string`の実装によるが、多くのコンパイラが対応している
