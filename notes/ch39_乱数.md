# 第39章: 乱数

## 要点

### 疑似乱数の3層構造

C++ の `<random>` は **3つの部品の組み合わせ** で乱数を作る設計。

```
random_device → エンジン（mt19937）→ 分布（distribution）→ 欲しい値
   シード源       生の乱数を生成      範囲・形を整える
```

| 部品 | 役割 |
|---|---|
| **乱数源（`random_device`）** | OS から非決定的なエントロピーを得る。シード生成専用 |
| **エンジン（`mt19937`）** | 高速に疑似乱数を量産する |
| **分布（`xxx_distribution`）** | 「1〜6 の整数」「平均0の正規分布」などに整形 |

### 推奨パターン

```cpp
#include <random>

std::random_device rd;                            // 真の乱数源（遅い）
std::mt19937 engine(rd());                        // 疑似乱数エンジンに種を渡す
std::uniform_int_distribution<int> dist(1, 6);    // 1〜6 の整数

int dice = dist(engine);                           // サイコロを振る
```

### シードとは

**疑似乱数の初期値（種）**。コンピュータの乱数は計算で次々と数を作る疑似乱数で、その出発点になるのがシード。

```
シード(42) → 1804289383 → 846930886 → 1681692777 → ...
シード(99) → 670204628  → 53728733  → 2138253845 → ...
```

**同じシードなら必ず同じ乱数列が出る** のが疑似乱数の最重要な性質。

```cpp
std::mt19937 a(42);
std::mt19937 b(42);
// a と b は完全に同じ乱数列を生成する
```

### 毎回違う乱数が欲しいとき

固定シードだと毎回同じ結果。実行ごとに違う結果が欲しいなら、**実行時に変動する値** をシードにする。

```cpp
// パターン1: random_device（推奨）
std::random_device rd;
std::mt19937 engine(rd());

// パターン2: 現在時刻
auto seed = std::chrono::system_clock::now().time_since_epoch().count();
std::mt19937 engine2(seed);
```

### シードを固定する用途（再現性）

「毎回違う乱数」だけが正解ではない。**わざと固定する** 場面も多い。

| 用途 | シード |
|---|---|
| ゲームの本番プレイ | ランダム（`random_device`） |
| 「シード値で同じ世界を共有」（Minecraft 等） | 固定 |
| 機械学習・科学計算の再現実験 | 固定 |
| 単体テスト・デバッグ | 固定（バグが再現する） |

```cpp
TEST(RandomTest, Distribution) {
    std::mt19937 engine(12345);   // 固定 → 結果が決定的
    // ...
}
```

### 分布（distribution）の役割

エンジンだけだと巨大な整数が出るだけ。**分布が範囲・形を整える**。

### 実務で使う分布（覚える優先度高）

| 分布 | 用途 |
|---|---|
| `uniform_int_distribution<int>(1, 6)` | 整数の一様（サイコロ、添字） |
| `uniform_real_distribution<double>(0.0, 1.0)` | 実数の一様、確率判定 |
| `bernoulli_distribution(0.3)` | 30% の確率で `true` |

これだけで日常コードの9割をカバーできる。

### 名前だけ知っておくと便利な分布

| 分布 | いつ使うか |
|---|---|
| `normal_distribution` | ノイズ生成、シミュレーション、機械学習 |
| `discrete_distribution` | 重み付きランダム選択（ガチャの確率テーブル） |

### 残りの分布は深追い不要

`poisson` / `exponential` / `gamma` / `chi_squared` / `student_t` / `fisher_f` / `weibull` / `cauchy` / `lognormal` / `binomial` / `negative_binomial` / `geometric` …

これらは **統計・数値計算・科学技術系の専門領域** 用。一般的なアプリ開発では出番がない。**「こういう分布があるらしい」と認識して、必要になったら調べる** で十分。

### 古い `rand()` との違い

C 言語由来の `rand()` / `srand()` は **現代 C++ では非推奨**。

```cpp
// 古いスタイル（避ける）
#include <cstdlib>
#include <ctime>
srand(time(nullptr));
int x = rand() % 100;
```

問題点：
- 品質が低い（ビットの偏り）
- スレッドセーフでない
- グローバル状態（複数の独立した乱数源を持てない）
- `% N` で範囲指定すると分布が偏る

`<random>` を使えばすべて解消。

## C#との比較

| 項目 | C# | C++ |
|---|---|---|
| 標準乱数クラス | `System.Random` | `<random>`（エンジン＋分布） |
| 設計 | 1クラスで全部 | **3層構造**（源・エンジン・分布） |
| シード指定 | `new Random(42)` | `mt19937 engine(42)` |
| シードなし | 自動（時刻ベース） | `random_device` を明示 |
| 暗号品質 | `RandomNumberGenerator.Create()` | `std::random_device` |
| 範囲指定 | `Next(min, max)` | `uniform_int_distribution` |
| 分布の種類 | **一様のみ**（正規分布等は自前計算） | **多数標準装備** |
| スレッド安全性 | `Random.Shared`（.NET 6+） | `thread_local` で自前 |

C# の `Random` は「クラス1つで全部」設計でシンプル。
C++ は **「エンジン × 分布」** を組み合わせる設計で、書く量は多いが柔軟。

```csharp
// C#（簡潔）
var rng = new Random();
int x = rng.Next(1, 101);
```

```cpp
// C++（明示的・3層）
std::random_device rd;
std::mt19937 engine(rd());
std::uniform_int_distribution<int> dist(1, 100);
int x = dist(engine);
```

### C# にない便利機能

C++ の `<random>` には正規分布、指数分布、ベルヌーイ分布などが標準で揃っている。C# では自分で計算するか、外部ライブラリ（`MathNet.Numerics` など）が必要。

## 補足

### `std::random_device` の落とし穴

`random_device` は **実装依存**。古い MinGW では決定的（毎回同じ値）という不具合があった。

回避策：

```cpp
auto t = std::chrono::high_resolution_clock::now().time_since_epoch().count();
std::mt19937 engine(static_cast<unsigned>(t));
```

最近の MSYS2 / MinGW は修正済みだが、移植性を気にするなら覚えておく。

### シード空間と衝突

`mt19937` の状態空間は約 $2^{19937}$ ビットだが、シードに渡せるのは `unsigned int`（32ビット）→ 約43億通りしか初期状態を選べない。

並列実行で衝突を避けたいときは `std::seed_seq` で複数のエントロピーを混ぜる：

```cpp
std::seed_seq seq{rd(), rd(), rd(), rd()};
std::mt19937 engine(seq);
```

### マルチスレッドでの注意

乱数エンジンは状態を持つ → スレッドごとに別のエンジンを用意するのが安全。

```cpp
thread_local std::mt19937 engine{std::random_device{}()};

int generate() {
    std::uniform_int_distribution<int> dist(1, 100);
    return dist(engine);
}
```

`thread_local` でスレッドごとに独立した乱数源を持てる。

### エンジンの選び方

| エンジン | 速度 | 品質 | 状態サイズ | 用途 |
|---|---|---|---|---|
| `std::minstd_rand` | 速い | 普通 | 小 | 軽量 |
| `std::mt19937` | 速い | 良い | 大（2.5KB） | **デフォルト** |
| `std::mt19937_64` | 速い | 良い | 大 | 64ビット |
| `std::ranlux24/48` | 遅い | 高品質 | 中 | 数値計算 |

**迷ったら `mt19937`**。

### 暗号目的には使うな

`mt19937` は統計的に良質だが **暗号には使えない**。状態を観察すれば次の値が予測できる。

暗号目的（パスワード生成、トークン、鍵）には：
- C++ には標準の暗号品質乱数はない
- OS 提供の API（Windows: `BCryptGenRandom`、Linux: `/dev/urandom`、`getrandom()`）を使う
- ライブラリ（OpenSSL の `RAND_bytes` 等）

### 分布の使い分けの考え方

「**この分布で何が出るか**」を覚える必要はない。**「こういう状況にはこの分布」** と紐付けるのが実用的。

| 状況 | 分布 |
|---|---|
| 1〜N の中から1つ | `uniform_int_distribution` |
| 0.0〜1.0 の確率値 | `uniform_real_distribution` |
| 「成功 or 失敗」の試行 | `bernoulli_distribution` |
| 平均周辺にばらつかせたい | `normal_distribution` |
| 重み付き選択 | `discrete_distribution` |

### 分布のリセット

分布も内部状態を持つことがある。完全にリセットしたいときは `.reset()`：

```cpp
std::normal_distribution<double> dist(0.0, 1.0);
dist(engine);
dist.reset();   // 内部状態をクリア
```

`uniform_int_distribution` などは状態を持たないので `.reset()` の効果はない。

### 実用例

```cpp
// シャッフル
std::vector<int> v = {1,2,3,4,5};
std::mt19937 engine(std::random_device{}());
std::shuffle(v.begin(), v.end(), engine);

// 重み付き選択（ガチャ）
std::discrete_distribution<int> gacha({70, 25, 4, 1});  // R/SR/SSR/UR
int rarity = gacha(engine);

// 30% の確率で発生
std::bernoulli_distribution chance(0.3);
if (chance(engine)) { /* 発生 */ }
```

## 理解度

- [x] シード（疑似乱数の初期値）の役割
- [x] 同じシードなら同じ乱数列が出る
- [x] `random_device` / エンジン / 分布の3層構造
- [x] `random_device` はシード生成専用、`mt19937` が量産
- [x] `mt19937` がデフォルト選択
- [x] 毎回違う乱数を出すには `random_device` を使う
- [x] 固定シードが必要な場面（テスト、再現実験、デバッグ）
- [x] 分布の役割（範囲・形を整える）
- [x] 実務で頻出する3つ：`uniform_int` / `uniform_real` / `bernoulli`
- [x] 残りの分布は深追いせず、必要になったら調べる方針
- [x] `rand()` を使わず `<random>` を使う
- [x] スレッドごとに `thread_local` でエンジンを持つ
- [x] 暗号目的には `mt19937` を使わない
- [x] C# の `Random`（1クラス完結・一様のみ）との違い

## 次章へ

乱数の3層設計を押さえたので、次は **継承とポリモーフィズム**（基底クラス、`virtual`、仮想デストラクタ）や、**例外処理の続き**、あるいは **時間関連（`<chrono>`）** など実用ライブラリの章に進む可能性がある。`<random>` で `random_device` や `chrono` が登場したので、これらの周辺ライブラリの章が続くかもしれない。
