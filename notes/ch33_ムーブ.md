# 第33章: ムーブ

## 要点

### ムーブの本質：所有権の譲渡

ムーブは **「オブジェクトが所有しているヒープ上のリソースを、別のオブジェクトに譲渡する」** 操作。コピーのように中身を複製するのではなく、ポインタの付け替えだけで済ませる。

```cpp
std::vector<int> v1 = {1, 2, 3, 4, 5};
std::vector<int> v2 = v1;              // コピー：ヒープ領域を複製（重い）
std::vector<int> v3 = std::move(v1);   // ムーブ：ポインタだけ譲渡（軽い）
// v1 は「有効だが未規定の状態」（通常は空）になる
```

### 図解：コピー vs ムーブ

```
コピー：
v1                v2
┌────────┐      ┌────────┐
│ ptr ───┼──┐   │ ptr ───┼──┐
└────────┘  │   └────────┘  │
            ↓               ↓
     ┌───────────┐   ┌───────────┐
     │ [1,2,3]   │   │ [1,2,3]   │  ← 別々に確保
     └───────────┘   └───────────┘

ムーブ：
v1                v3
┌────────┐      ┌────────┐
│ ptr: ∅ │      │ ptr ───┼──┐
└────────┘      └────────┘  │
                            ↓
                     ┌───────────┐
                     │ [1,2,3]   │  ← 元の領域をそのまま
                     └───────────┘
```

### `std::move` の正体

**実は何もムーブしない**。単に lvalue を rvalue にキャストするだけ。

```cpp
T&& std::move(T& x);   // 概念的にはこれだけ
```

rvalue に変換されることで、オーバーロード解決がムーブコンストラクタ／ムーブ代入を選ぶようになる、というのが本当の動作。

### ムーブコンストラクタとムーブ代入演算子

```cpp
class MyArray {
    int* data;
    size_t size;
public:
    // ムーブコンストラクタ（rvalue参照 && で受ける）
    MyArray(MyArray&& other) noexcept
        : data(other.data), size(other.size) {
        other.data = nullptr;   // ← 重要：元を空にする
        other.size = 0;
    }

    // ムーブ代入演算子
    MyArray& operator=(MyArray&& other) noexcept {
        if (this != &other) {
            delete[] data;           // 既存を解放
            data = other.data;       // ポインタを奪う
            size = other.size;
            other.data = nullptr;    // 元を空に
            other.size = 0;
        }
        return *this;
    }

    ~MyArray() { delete[] data; }
};
```

ポイント：
- **ポインタを奪う** + **元を `nullptr` にする**の2ステップ
- 元を `nullptr` にしないと、デストラクタで二重解放になる

### `noexcept` が重要

ムーブコンストラクタ／代入演算子には `noexcept` を付ける。

```cpp
MyArray(MyArray&& other) noexcept { ... }
```

理由：`std::vector` などの標準コンテナは、ムーブが `noexcept` でないと**コピーにフォールバックする**（例外安全のため）。せっかくのムーブが効かなくなる。

### ムーブ後の状態：有効だが未規定

```cpp
std::string s1 = "hello";
std::string s2 = std::move(s1);
// s1 は valid but unspecified state
```

| ムーブ後にやっていいこと | やってはいけないこと |
|---|---|
| デストラクタ実行（スコープ抜け） | 値を読む（`cout << s1` など） |
| 新しい値の代入（`s1 = "new"`） | 前提条件のある操作（`v.front()` など） |
| `size()`, `empty()`, `clear()` など状態非依存の操作 | `operator[]`, `front()`, `back()` |

**実務の指針**：ムーブ後は「再代入するかスコープを抜けて捨てる」までは触らない。

### 基本型・クラスのデフォルトムーブは「コピーと同じ」

```cpp
int a = 10;
int b = std::move(a);
// b は 10、a も 10（何も動いていない）
```

理由：`int` は値そのもの。譲渡すべきリソース（ヒープ上のデータ）が存在しない → コピーするしかない。

クラスのデフォルトムーブは「メンバーを1つずつムーブする」だけなので、メンバーが全部 `int` などの基本型なら結果的にコピーと同じ。

```cpp
class Point {
    int x, y;
    // コンパイラ自動生成 → x, y をムーブ → 実質コピー
};
```

### ムーブが本当に効くのは「ヒープリソースを持つ」とき

```cpp
class Container {
    std::vector<int> data;   // ← ムーブで真価を発揮
    int count;               // ← ムーブしてもコピーと同じ
};
```

`std::vector`, `std::string`, `std::unique_ptr` などがメンバーにいると、ムーブに意味が出てくる。

### Rule of Five

Rule of Three にムーブが加わって5つに：

1. デストラクタ
2. コピーコンストラクタ
3. コピー代入演算子
4. **ムーブコンストラクタ**
5. **ムーブ代入演算子**

1つでも書くなら5つ全部書く、が原則。ただし現代では **Rule of Zero**（全部書かずに STL に任せる）が推奨。

### コピーと同じポインタを所有する問題（復習）

```cpp
class MyArray {
    int* data;
public:
    MyArray(size_t n) : data(new int[n]) {}
    ~MyArray() { delete[] data; }
    // コピー定義なし → デフォルトは浅いコピー
};

MyArray a(10);
MyArray b = a;   // 浅いコピー：a と b が同じヒープ領域を指す
                 // スコープ終了時に二重解放でクラッシュ
```

### 対策の5つの選択肢

| 対策 | 使う場面 |
|---|---|
| ディープコピーを実装 | コピーに意味がある場合 |
| ムーブで所有権を移す | コピーせず譲渡したい |
| コピー禁止（`= delete`） | コピーが意味をなさない場合 |
| `std::unique_ptr` | 単独所有 |
| `std::shared_ptr` | 共有所有（参照カウント） |

## C#との比較

| 項目 | C# | C++ |
|---|---|---|
| ムーブの概念 | 基本的に存在しない | 言語機能として存在 |
| 参照型の代入 | 参照コピー（軽い） | 値コピー（重い）/ ムーブで回避 |
| 所有権の明示 | なし（GC管理） | `unique_ptr` / `shared_ptr` で表現 |
| リソース解放 | GC（非決定論的） | デストラクタ（決定論的） |
| 二重解放 | 構造的に起こらない | 注意しないと起こる |
| `=` の意味 | 参照のコピー | **オブジェクトの複製**（コピーorムーブ） |
| 確定解放の手段 | `IDisposable` + `using` | デストラクタ（スコープ終了で自動） |
| GCタイミング制御 | `GC.Collect()`（非推奨） | そもそも GC がない |

## 補足

### スタックとヒープの特徴（前提知識）

ムーブを理解するにはメモリの基本が必要。

| 観点 | スタック | ヒープ |
|---|---|---|
| 管理 | 自動（コンパイラ） | 手動（`new`/`delete`、RAII で自動化可） |
| 速度 | 非常に速い | 遅い |
| サイズ | 小（1〜8MB） | 大（GB単位） |
| ライフタイム | スコープ依存 | `delete` まで |
| 配置される例 | ローカル変数、関数引数 | `new`した物、vector の中身 |

```cpp
void example() {
    int a = 10;                      // スタック
    int* p = new int(20);            // p: スタック、*p: ヒープ
    std::vector<int> v = {1,2,3};    // v本体: スタック、中身: ヒープ
    delete p;
}  // スコープ終了：スタック自動解放、v のデストラクタで中身も解放
```

### ムーブの対象はヒープ上のリソース

- スタック上の値（`int` など）はそこに値が埋まっている → 動かせない
- ヒープ上のデータはポインタ経由 → ポインタを付け替えて譲渡できる＝ムーブ

**「場所」ではなく「構造」が重要**：vector 本体がスタックにあっても、中のデータは必ずヒープにあるのでムーブが効く。

### C#で二重解放が起きない理由

C#はGCが「誰からも参照されなくなったオブジェクトを、あとで1回だけ解放する」方式。

```csharp
var a = new MyClass();
var b = a;       // 参照コピー（同じオブジェクト）
a = null;
b = null;
// 両方 null になったら、GC が「いずれ」1回だけ解放
```

- C++：各オブジェクトがスコープ終了時に独立に解放 → 衝突する
- C#：GC が全体を俯瞰して1回だけ解放 → 衝突しようがない

### C#のGCタイミングは基本的に制御できない

- `GC.Collect()` で強制実行は可能だが原則非推奨（パフォーマンス悪化）
- `GC.TryStartNoGCRegion()` で一時的な抑制は可能
- 決定論的に解放したいリソース（ファイル、DB接続）は `IDisposable` + `using` を使う

```csharp
using var stream = new FileStream(...);
// スコープ終了で確実に Dispose（メモリ自体はGC管轄）
```

C#は「メモリ（GC管轄）」と「リソース（IDisposable管轄）」で扱いが分かれている。
C++は RAII でどちらも統一的に扱える。

### `std::shared_ptr` は「ミニGC」

```cpp
auto p1 = std::make_shared<int>(10);
auto p2 = p1;   // シャローコピーだが安全、参照カウント: 2
// 最後の所有者がスコープを抜けたときに1回だけ解放
```

C#の参照型に最も近い動作。ただしC++では**必要な箇所だけ選んで使う**のが違い。循環参照には `weak_ptr` が必要（GCのように自動検出されない）。

### ディープコピーはヒープ領域の複製

```cpp
Buffer(const Buffer& other) : size(other.size) {
    data = new int[size];                            // ① ヒープに新しく確保
    std::copy(other.data, other.data + size, data);  // ② 中身を複製
}
```

- **① 新たにヒープ領域を確保**
- **② 元の中身をそこにコピー**

これでコピー元・先が別々のヒープ領域を持つ → 独立。

### 標準ライブラリの所有権戦略

| クラス | コピー時の動作 |
|---|---|
| `std::vector`, `std::string`, `std::map` など | ディープコピー |
| `std::unique_ptr`, `std::thread`, `std::mutex` | コピー禁止（ムーブのみ） |
| `std::shared_ptr` | 参照カウントで共有 |

「所有権を持つ標準クラスは、ディープコピーかコピー禁止のどちらか」。中途半端なシャローコピーのデフォルトは避けられている。

### コピー可能＋ムーブ最適化のパターン

```cpp
std::vector<int> makeLarge() {
    std::vector<int> v(1000000);
    return v;              // RVO or ムーブで最適化、コピーされない
}

auto x = makeLarge();      // 100万要素だが高速
```

「コピーが可能＋ムーブも可能」な場合、コンパイラや標準ライブラリが自動で安い方（ムーブ or コピー省略）を選んでくれる。

### ムーブを使うべき場面・使うべきでない場面

**使うべき**：
- 関数の引数に大きなオブジェクトを「譲渡」したい
- コンテナに一時オブジェクトを入れる（`push_back(std::move(x))`）
- 所有権を明確に移したい

**使わないほうがいい**：
- 戻り値の `return` で `std::move` を書く（RVO を阻害する）
- `const` 変数への `std::move`（実質コピーになる）
- 基本型（`int` など）への `std::move`（無意味）

### Rule of Zero（現代C++の推奨）

```cpp
class Buffer {
    std::vector<int> data;   // vector が全部面倒見てくれる
    // デストラクタ、コピー、ムーブ全部書かなくていい
};
```

生ポインタで所有権を持たず、`std::vector` / `std::string` / `std::unique_ptr` / `std::shared_ptr` に任せれば、コピー・ムーブ・デストラクタが自動で正しく動く。

## 理解度

- [x] ムーブの本質は所有権の譲渡（コピーとの違い）
- [x] `std::move` は単なる rvalue へのキャスト
- [x] ムーブコンストラクタ／代入演算子の実装（ポインタ奪取 + 元を nullptr）
- [x] `noexcept` を付ける理由（標準コンテナのフォールバック回避）
- [x] ムーブ後のオブジェクトは「有効だが未規定」、触らないのが基本
- [x] 基本型のデフォルトムーブはコピーと同じ（所有リソースがないため）
- [x] ムーブが効くのはヒープリソースを持つメンバーがいる場合
- [x] スタックとヒープの違いと役割
- [x] シャローコピーで二重解放が起きる仕組み
- [x] 対策：ディープコピー／ムーブ／コピー禁止／スマートポインタ
- [x] C#でこの問題が起きない理由（GCによる集中管理）
- [x] C#のGCタイミングは基本制御できず、リソース解放は `IDisposable` 担当
- [x] `std::shared_ptr` が「ミニGC」的に動く
- [x] Rule of Five と Rule of Zero
