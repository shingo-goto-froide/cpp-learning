# 第42章: デバッガ

## 要点

### デバッガとは

実行中のプログラムを **任意の場所で止めて、変数の値を見たり一行ずつ進めたりできるツール**。書籍は GDB（コマンドラインのデバッガ）を扱うが、実務では IDE の GUI デバッガで完結することが多い。

### GDB と VSCode の関係

**VSCode のデバッガは内部で GDB を動かしている**。GUI でボタンを押すと、裏で対応する GDB コマンドが送られる。

```
VSCode（GUI、ボタン）
   ↓ 命令を送る
GDB（実行エンジン）
   ↓
プログラム
```

操作方法が違うだけで本質は同じ。GDB のコマンドを覚えなくても VSCode で同じことができる。

### デバッグビルドのためのコンパイラオプション

#### `-g` ― デバッグ情報を埋め込む

```bash
g++ -g main.cpp -o app
```

`-g` がないとブレークポイントが効かない／変数名が見えない。デバッグ時は必須。

#### `-O0` ― 最適化を切る

```bash
g++ -g -O0 main.cpp -o app
```

`-O2` などの最適化が効いていると、変数が消えたり実行順が飛んだりして **追跡できなくなる**。デバッグビルドでは最適化を切るのが定石。

| ビルド種別 | 推奨フラグ |
|---|---|
| デバッグ | `-g -O0` |
| リリース | `-O2` または `-O3`、`-DNDEBUG` |

### 主なデバッガ操作

| 機能 | VSCode | GDB |
|---|---|---|
| ブレークポイント | 行番号左をクリック | `break main.cpp:10` |
| 実行 / 続行 | F5 | `run` / `continue` |
| ステップオーバー | F10（行単位で進む） | `next` |
| ステップイン | F11（関数の中に入る） | `step` |
| ステップアウト | Shift+F11（関数から出る） | `finish` |
| 変数の値を見る | Variables パネル／ホバー | `print x` |
| ウォッチ式 | Watch パネルに追加 | `watch x` |
| 条件付きブレーク | ブレークポイント右クリック → Edit | `break ... if x > 10` |
| コールスタック表示 | Call Stack パネル | `backtrace`（`bt`） |
| プログラム中断 | Pause ボタン | Ctrl+C |

### ステップ操作の使い分け

```cpp
int main() {
    int x = compute();   // ← ここで停止中
    int y = process(x);
    return 0;
}
```

- **ステップオーバー（F10）**：`compute()` の中には入らず、次の行へ
- **ステップイン（F11）**：`compute()` の中に入って一行ずつ追う
- **ステップアウト（Shift+F11）**：今の関数を最後まで実行して呼び出し元に戻る

中身が信頼できる関数（標準ライブラリ等）は **オーバー**、自分のコードで挙動を確認したいときは **イン**。

### 条件付きブレークポイント

「ループの100回目で止めたい」「特定の値のときだけ止めたい」ときに使う。

```cpp
for (int i = 0; i < 1000; ++i) {
    process(i);   // ← i == 100 のときだけ止めたい
}
```

VSCode：ブレークポイントを右クリック → Edit Breakpoint → `i == 100` と入力。

### ウォッチ式

特定の式を **常に監視** する。変数だけでなく式も書ける。

```
x + y
arr[i].value
ptr != nullptr
```

ステップ実行のたびに値が更新されて表示される。

### `launch.json`（VSCode）

VSCode のデバッグ設定ファイル。実行ファイル、引数、作業ディレクトリ、環境変数などを指定する。

```json
{
    "version": "0.2.0",
    "configurations": [
        {
            "name": "Debug C++",
            "type": "cppdbg",
            "request": "launch",
            "program": "${workspaceFolder}/app.exe",
            "args": [],
            "cwd": "${workspaceFolder}",
            "stopAtEntry": false,
            "MIMode": "gdb"
        }
    ]
}
```

C# の `.vscode/launch.json` と同じ位置・同じ役割。

### コールスタックを読む

クラッシュやブレーク時、**呼び出しの履歴** が左に表示される。

```
[現在] foo()       ← ここで止まった
       bar()       ← foo を呼んだ
       main()      ← bar を呼んだ
```

クリックすると各フレームの変数を確認できる。「どこから来たか」を遡れるのでバグの原因追跡に重要。

### 典型的なデバッグの流れ

1. **再現**：バグが起きるシナリオを特定
2. **疑わしい場所にブレーク**：怪しい関数の入口、変数の初期化箇所
3. **ステップ実行**：一行ずつ進めて変数の変化を見る
4. **ウォッチで監視**：特定の式が想定外の値になる瞬間を捉える
5. **コールスタック確認**：誰がこの関数を呼んだか確認

### `printf` デバッグとの使い分け

| 手段 | 向いている場面 |
|---|---|
| `printf` / `std::cerr` | ループの大量出力、再現性の低いバグ、本番環境 |
| デバッガ | 1回の実行で詳細を追いたい、変数の状態を細かく見たい |

両方使えると最強。再現が難しいバグはログ、再現できればデバッガ、と使い分ける。

## C#との比較

| 項目 | C# | C++ |
|---|---|---|
| 主要デバッガ | Visual Studio / VSCode 標準 | GDB / LLDB / MSVC デバッガ |
| 操作方法 | F5 / F10 / F11 | 同じ |
| デバッグビルド | プロジェクト設定で自動 | `-g -O0` を明示的に指定 |
| 最適化の影響 | あまり気にしない | `-O2` 以上だと変数が追えなくなる |
| シンボル情報 | `.pdb` ファイル | バイナリに `-g` で埋め込み or `.dwarf` |
| ホットリロード | あり（C# 側で進化中） | 一般的でない |
| 例外時の停止 | First-chance / unhandled で自動 | 設定で有効化 |

操作感は同じ（VSCode を使えば違いを意識しないレベル）。違うのは **「デバッグビルドを自分で作る必要がある」** 点。C# は `dotnet build -c Debug` で自動だが、C++ は `g++ -g -O0` を明示する。

```csharp
// C#（普段意識しない）
// プロジェクトの Debug 構成を選んで F5、それだけ
```

```bash
# C++（明示的に指定）
g++ -g -O0 main.cpp -o app
# VSCode の launch.json で app を指定して F5
```

## 補足

### GDB のコマンドチートシート

VSCode で完結するなら不要だが、リモートサーバーや CI 環境で `.exe` だけ動かすときに使う場面がある。

| GDB コマンド | 効果 |
|---|---|
| `break main.cpp:10` | 10行目にブレーク |
| `break foo` | 関数 foo の入口にブレーク |
| `info breakpoints` | ブレーク一覧 |
| `delete N` | N番目のブレーク削除 |
| `run [args]` | 実行開始 |
| `continue`（`c`） | 続行 |
| `next`（`n`） | ステップオーバー |
| `step`（`s`） | ステップイン |
| `finish` | ステップアウト |
| `print x`（`p x`） | 変数表示 |
| `display x` | 自動表示（毎ステップ） |
| `backtrace`（`bt`） | コールスタック |
| `frame N` | N番目のフレームに移動 |
| `quit`（`q`） | 終了 |

### コアダンプ

クラッシュ時のメモリ状態を保存したファイル。後から GDB で開いて原因調査ができる。

```bash
ulimit -c unlimited       # コアダンプ有効化（Linux）
./app                     # クラッシュ → core ファイル生成
gdb ./app core            # 解析
```

Windows では「ミニダンプ（`.dmp`）」が同じ役割。Visual Studio で開ける。

### Sanitizer（実行時バグ検出）

GDB と併用すると強力な、ビルド時に有効化する **動的解析ツール**。

```bash
g++ -fsanitize=address -g main.cpp     # メモリエラー検出
g++ -fsanitize=undefined -g main.cpp   # 未定義動作検出
g++ -fsanitize=thread -g main.cpp      # データ競合検出
```

| Sanitizer | 検出するもの |
|---|---|
| AddressSanitizer (ASan) | バッファオーバーラン、use-after-free、リーク |
| UndefinedBehaviorSanitizer (UBSan) | 整数オーバーフロー、null 参照、配列範囲外 |
| ThreadSanitizer (TSan) | データ競合 |

実行時に異常を検出してメッセージを出してくれる。デバッガ単体では気付けない種類のバグに有効。

### `assert`

実行時の前提条件チェック。デバッグビルドで `false` ならプログラムを止める。

```cpp
#include <cassert>
assert(x > 0);   // x <= 0 なら abort
```

リリースビルド（`-DNDEBUG`）では何もしない。**契約による設計** の手段。

### `static_assert`（コンパイル時）

コンパイル時にチェックする `assert`。条件が偽ならビルドエラー。

```cpp
static_assert(sizeof(int) == 4, "int must be 4 bytes");
```

実行前に問題を発見できる。

### デバッグログの工夫

```cpp
#define LOG(x) std::cerr << "[" << __FILE__ << ":" << __LINE__ \
                         << " " << __func__ << "] " << x << "\n"

LOG("entering loop, n=" << n);
```

`__FILE__` / `__LINE__` / `__func__` を使うとログから場所を特定しやすい。

### リモートデバッグ

サーバー上のプログラムを手元の VSCode で操作するパターン。

```
[手元 VSCode] ──→ [SSH／TCP] ──→ [リモート GDB] ──→ [リモートプログラム]
```

`launch.json` で `pipeTransport` や `gdbserver` を設定する。組み込み開発や Docker コンテナ内のアプリのデバッグに使う。

### よくある詰まりポイント

#### ブレークが効かない

- `-g` を付けてビルドしたか
- 最適化を切っているか（`-O0`）
- ビルド済みバイナリと実行バイナリが一致しているか（再ビルド忘れ）

#### 変数が `<optimized out>` と表示される

- 最適化が効いている → `-O0` で再ビルド

#### 実行ファイルとソースが対応していない

- 古いバイナリで動かしている → クリーンビルド
