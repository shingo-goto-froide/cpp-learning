# 第40章: プリプロセッサ

## 要点

### プリプロセッサとは

ソースコードがコンパイラに渡される **前** に、テキスト処理を行う仕組み。`#` で始まる行が処理対象。

```
ソースコード → [プリプロセッサ] → 展開済みコード → [コンパイラ] → 機械語
```

プリプロセッサが処理した結果がコンパイラに渡される。**コンパイラには `#` で始まる行は届かない**。

### 主なディレクティブ

| ディレクティブ | 用途 |
|---|---|
| `#include` | ヘッダの取り込み |
| `#pragma once` | 多重 include 防止 |
| `#define` / `#undef` | マクロ定義／削除 |
| `#ifdef` / `#ifndef` / `#endif` | 条件付きコンパイル（定義の有無） |
| `#if` / `#elif` / `#else` | 条件付きコンパイル（値での比較） |
| `#error` / `#warning` | コンパイル時メッセージ |
| `#pragma` | コンパイラ独自の指示 |

### `#include` の動作と多重 include 問題

`#include` は **指定したファイルの中身をその場にテキスト展開する** だけ。

```cpp
#include <iostream>     // 標準ライブラリ（< > で囲む）
#include "myheader.hpp" // 自作ヘッダ（" " で囲む）
```

| 形式 | 探し方 |
|---|---|
| `<...>` | 標準ライブラリのパスから |
| `"..."` | カレントディレクトリ → 標準パス |

問題：同じヘッダが複数回展開されるとクラスや関数の **多重定義エラー** になる。

```cpp
// foo.hpp
class Foo { int x; };

// main.cpp
#include "foo.hpp"
#include "foo.hpp"   // ❌ class Foo が2回定義 → エラー
```

直接2回書くことは少ないが、**ヘッダ同士の依存で間接的に多重 include される** ケースは頻発する。

```cpp
// a.hpp:    #include "common.hpp"
// b.hpp:    #include "common.hpp"
// main.cpp: #include "a.hpp"
//           #include "b.hpp"
// → common.hpp が2回展開されてエラー
```

### 解決策1: インクルードガード（伝統的）

ヘッダの中身をマクロで囲む。

```cpp
// foo.hpp
#ifndef FOO_HPP
#define FOO_HPP

class Foo { int x; };

#endif
```

仕組み：
1. 1回目の include：`FOO_HPP` 未定義 → 中身が展開され、`FOO_HPP` 定義
2. 2回目の include：`FOO_HPP` 定義済み → 中身は **空** として扱われる

### 解決策2: `#pragma once`（モダン）

1行で同じ効果。

```cpp
// foo.hpp
#pragma once

class Foo { int x; };
```

| 項目 | `#ifndef` ガード | `#pragma once` |
|---|---|---|
| 標準仕様 | ✅ 標準 | ❌ 非標準（事実上の標準） |
| 書く量 | 3行 | 1行 |
| マクロ名衝突 | あり得る | なし |
| 主要コンパイラ対応 | すべて | GCC/Clang/MSVC すべて |

**現代の主流**は `#pragma once`。移植性重視なら両方併記。

### 多重 include がエラーになる物・ならない物

```cpp
// ❌ エラー：定義系
class Foo { /* ... */ };       // クラス定義
struct S { /* ... */ };
enum E { A, B };
template<typename T> void f() {}

// ✅ 何度書いても OK：宣言のみ
extern int x;
void foo();

// ⚠️ 翻訳単位を超えるとリンクエラー：関数本体
void bar() { /* ... */ }       // ヘッダに書くなら inline 必須
```

### `#define` ― マクロ定義

#### 値ありマクロ（定数）

```cpp
#define PI 3.14159
#define MAX_SIZE 100

int arr[MAX_SIZE];     // → int arr[100];
```

現代では **`constexpr` を使うのが推奨**：

```cpp
constexpr double PI = 3.14159;       // 型安全
constexpr int MAX_SIZE = 100;
```

#### 関数風マクロ

```cpp
#define SQUARE(x) ((x) * (x))

int y = SQUARE(5);           // → ((5) * (5)) → 25
int z = SQUARE(a + 1);       // → ((a + 1) * (a + 1))
```

注意：括弧を **過剰なくらい付ける**。さもないと展開時に演算順がおかしくなる。

```cpp
#define BAD(x) x * x
BAD(a + 1);   // → a + 1 * a + 1 → 意図と違う計算
```

代替：`constexpr` 関数 / `inline` 関数 / テンプレート。

#### マクロの削除

```cpp
#undef MAX_SIZE
```

### 値なしマクロと `#ifdef`

`#define` は値なしでも使える。**「定義されているかどうか」だけを見たい場面** で使う。

```cpp
#define DEBUG          // 値なしで定義

#ifdef DEBUG           // 定義されている → true
    std::cout << "debug mode\n";
#endif
```

| ディレクティブ | 条件 |
|---|---|
| `#ifdef X` | `X` が定義されている |
| `#ifndef X` | `X` が定義されていない |

`#define DEBUG` でも `#define DEBUG 1` でも、`#ifdef` 的には **どちらも「定義されている」** で同じ。

### `#if` ― 値で分岐

値で分岐するときは `#if`。

```cpp
#define VERSION 2

#if VERSION == 2
    // VERSION が 2 のとき
#elif VERSION == 3
    // VERSION が 3 のとき
#else
    // それ以外
#endif
```

#### `defined()` 演算子

`#if` の中でマクロの有無を判定。論理演算と組み合わせられる。

```cpp
#if defined(_WIN32) && !defined(__MINGW32__)
    // Windows MSVC のみ
#endif

#if __cplusplus >= 202002L
    // C++20 以降
#endif
```

### マクロを定義する3つの方法

#### ① ソースコード内で `#define`

```cpp
#define DEBUG
#ifdef DEBUG
    // 有効
#endif
```

#### ② コンパイラのコマンドラインオプション `-D`（重要）

```bash
g++ -DDEBUG main.cpp           # #define DEBUG 相当
g++ -DDEBUG=1 main.cpp         # #define DEBUG 1 相当
g++ -DVERSION=2 main.cpp
g++ -DDEBUG -DVERBOSE main.cpp # 複数同時
```

`-D` は **「ソースの先頭に `#define` が書かれているのと同じ」** にするオプション。
**ソースを書き換えずに** ビルド構成を切り替えられるのが最大の利点。

```cpp
// main.cpp は編集しない
#ifdef DEBUG
    std::cerr << "debug log\n";
#endif
```

```bash
g++ -DDEBUG -O0 -g main.cpp -o app_debug      # デバッグビルド
g++         -O2    main.cpp -o app_release    # リリースビルド
```

MSVC は `/D`：

```bash
cl /DDEBUG main.cpp
```

#### ③ コンパイラ／環境が自動で定義

```cpp
#ifdef _WIN32       // Windows で自動定義
#ifdef __linux__    // Linux で自動定義
#ifdef __APPLE__    // macOS で自動定義
#if __cplusplus >= 202002L   // C++ 規格バージョン
```

### 定義済みマクロ（よく使う）

| マクロ | 内容 |
|---|---|
| `__FILE__` | 現在のファイル名 |
| `__LINE__` | 現在の行番号 |
| `__func__` | 現在の関数名 |
| `__cplusplus` | C++ 規格番号（例: `202302L` = C++23） |
| `__DATE__` / `__TIME__` | コンパイル日時 |
| `_WIN32` / `__linux__` / `__APPLE__` | プラットフォーム |
| `NDEBUG` | リリースビルド時に定義（`assert` 無効化） |

```cpp
std::cerr << __FILE__ << ":" << __LINE__ << " in " << __func__ << "\n";
```

### `#error` / `#warning`

コンパイル時にメッセージを出してビルドを止める／警告する。

```cpp
#if !defined(__cplusplus)
    #error "C++ コンパイラで使ってください"   // ビルド停止
#endif

#ifdef OLD_API
    #warning "OLD_API は非推奨です"           // 警告のみ
#endif
```

`#warning` は GCC/Clang の拡張だったが、C++23 で標準化された。

### `#pragma` ― コンパイラ独自の指示

意味は **コンパイラ依存**。代表例：

```cpp
#pragma once                     // 多重 include 防止

#pragma pack(push, 1)            // 構造体のパディングを1に（MSVC/GCC）
struct Packed { char a; int b; };
#pragma pack(pop)

#pragma GCC diagnostic ignored "-Wunused-variable"   // 警告抑制
```

知らない `#pragma` は無視される（仕様）。

### 実用的な使い方パターン

#### ① Debug／Release の切り替え（最頻出）

```cpp
#ifdef DEBUG
    std::cerr << "x = " << x << "\n";
#endif
```

リリースビルドではログ処理が **コードごと消える** → 性能影響ゼロ。

#### ② プラットフォーム別コード

```cpp
#if defined(_WIN32)
    #include <windows.h>
    void sleep_ms(int ms) { Sleep(ms); }
#elif defined(__linux__) || defined(__APPLE__)
    #include <unistd.h>
    void sleep_ms(int ms) { usleep(ms * 1000); }
#endif
```

普通の `if` だと、存在しない関数を呼ぼうとしてコンパイルエラー。`#ifdef` なら片方が **そもそもコンパイラに渡されない** ので問題が起きない。

#### ③ C++ バージョン分岐

```cpp
#if __cplusplus >= 202002L
    auto msg = std::format("x = {}", x);   // C++20
#else
    std::stringstream ss;
    ss << "x = " << x;
    auto msg = ss.str();
#endif
```

#### ④ assert の有無を切り替え

```cpp
#include <cassert>
assert(x > 0);   // NDEBUG が定義されていないとき → チェック実行
                 // 定義されているとき → 何もしない
```

#### ⑤ コードの一時無効化

`#if 0 ... #endif` はコメントアウトより安全（入れ子可能）。

```cpp
#if 0
    // 書き直し中なので一旦無効化
    foo();
    /* この中のコメントもぶつからない */
#endif
```

#### ⑥ デバッグログマクロ

```cpp
#ifdef DEBUG
    #define LOG(x) std::cerr << "[" << __FILE__ << ":" << __LINE__ << "] " << x << "\n"
#else
    #define LOG(x) ((void)0)
#endif

LOG("processing item " << i);
```

#### ⑦ ライブラリの DLL エクスポート（Windows）

```cpp
#ifdef BUILDING_MYLIB
    #define MYLIB_API __declspec(dllexport)
#else
    #define MYLIB_API __declspec(dllimport)
#endif

MYLIB_API void public_function();
```

### `#ifdef` vs `if constexpr`

C++17 以降、コンパイル時の分岐には `if constexpr` という現代的な手段もある。

```cpp
// プリプロセッサ（古典）
#ifdef DEBUG
    log("debug");
#endif

// if constexpr（現代）
constexpr bool DEBUG_MODE = true;
if constexpr (DEBUG_MODE) {
    log("debug");
}
```

| 項目 | `#ifdef` | `if constexpr` |
|---|---|---|
| 処理タイミング | コンパイル前のテキスト処理 | コンパイル時の言語機能 |
| 構文チェック | 読まれない部分は素通り | 型チェックは通る |
| 用途 | ビルド構成、プラットフォーム分岐 | 型による分岐、テンプレート |

ビルド構成は `#ifdef`、型ごとの分岐は `if constexpr`、と使い分ける。

### 確認方法

`-E` でプリプロセッサだけ動かして展開結果を確認できる。

```bash
g++ -DDEBUG -E main.cpp | head -20
```

`#error` で確認することもできる：

```cpp
#ifdef DEBUG
    #error "DEBUG is defined"
#endif
```

ビルドが失敗するなら定義されている。

## C#との比較

C# にもプリプロセッサ機能はあるが、**意図的に C++ より弱い**。

| 項目 | C# | C++ |
|---|---|---|
| `#include` | **なし**（アセンブリ参照で解決） | あり |
| `using System;` | 名前空間参照（コンパイラが解決） | `#include <iostream>`（テキスト展開） |
| `#define` | 可能（値なしのみ） | 可能（値あり／なし） |
| `#ifdef` | **なし** | あり |
| `#if SYMBOL` | 定義の有無を判定 | 値の比較も可 |
| 関数風マクロ | **なし** | あり |
| コマンドラインで定義 | `csc /define:DEBUG` | `g++ -DDEBUG` |
| `#region` / `#endregion` | あり | なし |
| `[Conditional("DEBUG")]` 属性 | あり（メソッド呼び出しの条件除去） | マクロで自前 |
| プラットフォーム分岐 | `.NET` が吸収 | `#ifdef` で自分で対応 |

C# は「**マクロは害**」という設計思想：
- マクロは値を持てない（`#define DEBUG` 単独のみ）
- 関数風マクロなし
- include はそもそも存在せず、アセンブリ参照で解決

C++ は歴史的経緯で多機能だが、**現代では `constexpr` / テンプレート / `inline` でマクロを置き換える** のが推奨。

```csharp
// C#
#if DEBUG
    Console.WriteLine("debug");
#endif
```

```cpp
// C++（同じ感覚で書ける）
#ifdef DEBUG
    std::cout << "debug\n";
#endif
```

C++ で違うのは **プラットフォーム分岐に `#ifdef` を使う** 点。C# は `.NET` が OS 差を吸収するので不要。

## 補足

### `inline` で多重定義を許す

ヘッダに関数の本体を書きたいときは `inline` を付けると、複数の翻訳単位に現れても OK になる。

```cpp
// foo.hpp
#pragma once

inline void greet() {
    std::cout << "hi\n";
}
```

`inline` は「速度のヒント」というより、**ODR（One Definition Rule）違反を許す宣言** という意味合いが強い。

### モジュール（C++20）

C++20 で導入された **モジュール** は、`#include` の問題を根本から解決する仕組み。

```cpp
// foo.cppm
export module foo;
export class Foo { int x; };

// main.cpp
import foo;
// ガード不要、テキスト展開ではない
```

C# の `using` に近い。普及はまだ進行中で、当面は `#include` + `#pragma once` が主流。

### 現代 C++ ではマクロを避ける方向

| 古い書き方 | 現代の代替 |
|---|---|
| `#define PI 3.14` | `constexpr double PI = 3.14;` |
| `#define MAX(a,b) ...` | `std::max(a, b)`（テンプレート） |
| `#define DEBUG` での条件コンパイル | `if constexpr` や設定フラグ |
| マクロでデバッグログ | `std::source_location`（C++20） |

ただし以下は現役で必須：
- `#include`
- `#pragma once`
- プラットフォーム判定の `#ifdef`
- ビルド構成切り替えの `#ifdef DEBUG`

### CMake/Makefile での `-D`

実務ではコマンドを直接打つよりビルドツールで管理する。

#### CMake

```cmake
add_executable(myapp main.cpp)

target_compile_definitions(myapp PRIVATE
    DEBUG
    VERSION=2
)
```

#### Makefile

```makefile
debug:
	g++ -DDEBUG -O0 -g main.cpp -o app

release:
	g++ -O2 main.cpp -o app
```

### ヘッダに書くもの・書かないもの

| 書く | 書かない（または `inline`） |
|---|---|
| クラス／構造体定義 | 関数の本体（`inline` なら OK） |
| テンプレート | 通常変数の定義（`inline` 変数なら OK） |
| `inline` 関数 | グローバル変数の `=` 付き定義 |
| `constexpr` 関数／変数 | |
| `enum` | |

ヘッダに書くと多重 include で多重定義になる物は、`inline` を付けるか `.cpp` に書く。

### ヘッダの命名

| 拡張子 | 意味 |
|---|---|
| `.h` | C 由来、C/C++ 両用に多い |
| `.hpp` | C++ 専用（一般的な慣習） |
| `.hxx` / `.hh` | あまり一般的でない |

標準ライブラリは拡張子なし（`<iostream>`, `<vector>`）。

### 実務での `#ifdef` の現実的な使いどころ

実用シーンの9割は次の2つ：

```cpp
// パターン1：デバッグログ
#ifdef DEBUG
    std::cerr << ... << "\n";
#endif

// パターン2：プラットフォーム分岐
#ifdef _WIN32
    // Windows 専用処理
#endif
```

これだけ覚えておけば、現代 C++ で `#ifdef` の使いどころで困ることはほぼない。

## 理解度

- [x] プリプロセッサはコンパイル前のテキスト処理
- [x] `#include` は中身をテキスト展開するだけ
- [x] 多重 include がエラーになる仕組み（クラス定義の重複）
- [x] `#pragma once` と `#ifndef` ガードの違いと使い分け
- [x] `#define` の2形態（値なし／値あり、定数／関数風）
- [x] 関数風マクロの括弧の重要性
- [x] 現代では `constexpr` でマクロを置き換える
- [x] `#ifdef` / `#ifndef` は「定義されているか」を判定
- [x] `#if` は値の比較ができる
- [x] `defined()` 演算子で複合条件を書ける
- [x] マクロ定義の3つの方法（ソース内 `#define` / `-D` / 自動）
- [x] `-D` でソースを変えずにビルド構成を切り替える
- [x] 定義済みマクロ（`__FILE__`, `__LINE__`, `__cplusplus` など）
- [x] `#error` / `#warning` の使い方
- [x] `#pragma` はコンパイラ依存
- [x] 実用パターン（Debug／Release、プラットフォーム分岐、バージョン分岐）
- [x] `if constexpr` との使い分け
- [x] C# にはない機能（`#include`, 関数風マクロ, `#ifdef`）
