# 第36章: スマートポインタ

## 要点

### スマートポインタとは

生ポインタ（`T*`）に **所有権管理機能** を追加したクラス。スコープを抜けると自動で `delete` してくれるので、メモリリークや二重解放を防げる。

C++11 以降、標準ライブラリには3種類のスマートポインタがある。

| 種類 | 所有モデル | コピー | ムーブ |
|---|---|:---:|:---:|
| `std::unique_ptr<T>` | 単独所有 | ❌ | ✅ |
| `std::shared_ptr<T>` | 共有所有（参照カウント） | ✅ | ✅ |
| `std::weak_ptr<T>` | 非所有の参照 | ✅ | ✅ |

### `std::make_unique<T>()` の動作

`make_unique<T>(args...)` は **2つのことを同時に行う**：

1. ヒープに `T` 型のオブジェクトを `new` する
2. そのアドレスを管理する `unique_ptr<T>` を作って返す

```cpp
auto p = std::make_unique<int>(0);

// 等価な手書きコード：
int* raw = new int(0);                  // ① ヒープ確保
std::unique_ptr<int> p(raw);            // ② unique_ptr に所有権委譲
```

### `<T>` と `()` の意味

```cpp
auto a = std::make_unique<int>(0);            // ヒープに int を作る、初期値 0
auto b = std::make_unique<std::string>("hi"); // ヒープに string を作る
auto c = std::make_unique<MyClass>(1, 2);     // MyClass のコンストラクタに 1, 2 を渡す
```

- `<T>` … 何のオブジェクトをヒープに作るか
- `()` … `T` のコンストラクタに渡す引数

### 中身へのアクセス

```cpp
auto p = std::make_unique<int>(0);
*p = 123;                          // 間接参照で値を書き換え

auto q = std::make_unique<MyClass>();
q->foo();                          // メンバアクセス
MyClass* raw = q.get();            // 中の生ポインタを取り出す（所有権はそのまま）
```

`*p` は **lvalue** なので代入できる（生ポインタと同じ感覚で使える）。

### `unique_ptr` はコピー禁止

```cpp
auto p = std::make_unique<int>(0);
auto q = p;             // ❌ コンパイルエラー
auto r = std::move(p);  // ✅ ムーブで所有権を譲渡（p は nullptr に）
```

「単独所有」を保証するため、コピーは `= delete` で禁止されている。もしコピーできたら、両者がスコープを抜けるときに二重解放してクラッシュする。

```cpp
template<typename T>
class unique_ptr {
public:
    unique_ptr(const unique_ptr&) = delete;            // コピー禁止
    unique_ptr& operator=(const unique_ptr&) = delete;
    unique_ptr(unique_ptr&&) noexcept;                 // ムーブはOK
    unique_ptr& operator=(unique_ptr&&) noexcept;
};
```

### 所有権の移動パターン

```cpp
// 関数に所有権を渡す
void take(std::unique_ptr<int> p) { /* ... */ }

auto p = std::make_unique<int>(42);
take(std::move(p));   // 所有権を関数に譲渡（以後 p は nullptr）

// 関数から所有権を受け取る
std::unique_ptr<int> make() {
    return std::make_unique<int>(42);   // ムーブで返る
}
auto p = make();
```

### `std::shared_ptr` — コピーできるスマートポインタ

複数箇所から **同じオブジェクトを共有** したいときに使う。内部の **参照カウント** で生存管理。

```cpp
auto p = std::make_shared<int>(42);   // カウント: 1
auto q = p;                           // カウント: 2
auto r = p;                           // カウント: 3
// p, q, r すべて同じヒープ領域を指す
```

### 参照カウントの仕組み

```
┌─────┐           ┌──────┐
│  p  ├──────────→│  42  │  ← データ
└─────┘    ┌─────→│count │  ← 制御ブロック（カウンタ）
┌─────┐    │      │  2   │
│  q  ├────┘      └──────┘
└─────┘
```

- コピーするたびにカウント +1
- スコープを抜けるたびにカウント −1
- **カウントが 0 になった瞬間に `delete`** を実行

```cpp
~shared_ptr() {
    if (--count == 0) {
        delete ptr;
        delete control_block;
    }
}
```

### 「最後の1人」が解放を担当

```cpp
auto p = std::make_shared<int>(42);  // count=1
{
    auto q = p;                       // count=2
    auto r = p;                       // count=3
}                                     // q, r 死亡 → count=1
// p のみ生存
// p がスコープを抜けたとき → count=0 → delete

// 最初に作ったのが p でも、最後まで生き残った人が解放する
```

`p` が一番最初に作られたから特別、ではない。**最後の1人が掃除当番**。

### `unique_ptr` vs `shared_ptr`

| 項目 | `unique_ptr` | `shared_ptr` |
|---|---|---|
| 所有者の数 | 1人だけ | 何人でも |
| コピー | ❌ 禁止 | ✅ OK |
| ムーブ | ✅ OK | ✅ OK |
| サイズ | ポインタ1個分 | ポインタ2個分＋制御ブロック |
| 速度 | 速い | カウント操作で少し遅い |
| スレッド安全性 | なし | カウント操作はアトミック |

**指針**：迷ったら `unique_ptr`。共有が必要なときだけ `shared_ptr`。

### `std::weak_ptr` — 循環参照を回避するための弱参照

`shared_ptr` だけだと **循環参照でリーク** する。

```cpp
struct Node {
    std::shared_ptr<Node> next;
};

auto a = std::make_shared<Node>();
auto b = std::make_shared<Node>();
a->next = b;   // a → b
b->next = a;   // b → a（循環！）
// 元の a, b が消えてもカウントが 0 にならない → 永久に解放されない
```

対策：循環の片方を `weak_ptr` にする。

```cpp
struct Node {
    std::shared_ptr<Node> next;
    std::weak_ptr<Node> prev;   // 弱参照（カウントを増やさない）
};

// アクセスするときは .lock() で shared_ptr に変換
if (auto sp = node.prev.lock()) {
    // sp が有効なときだけ処理
}
```

### `make_unique` / `make_shared` を使う理由

```cpp
// ❌ 古典的
std::unique_ptr<int> p(new int(42));
std::shared_ptr<int> q(new int(42));

// ✅ 推奨
auto p = std::make_unique<int>(42);
auto q = std::make_shared<int>(42);
```

利点：
- **例外安全**：複数引数の評価順問題を回避
- **`make_shared` は1回の確保で済む**：データ本体と制御ブロックを1つのメモリ確保にまとめる（性能向上）
- 型を1度書くだけで済む（DRY）

## C#との比較

C# の参照型は実質「すべてが `shared_ptr` のような動作」（GC管理）。

| C# | C++ で対応するもの |
|---|---|
| 通常の参照型（`new MyClass()`） | `std::shared_ptr<MyClass>`（に近い） |
| `IDisposable` + `using` | `std::unique_ptr<T>`（に近い、確定解放） |
| `WeakReference<T>` | `std::weak_ptr<T>` |
| GC が循環参照を自動回収 | **C++ は検出しない** → `weak_ptr` で手動回避 |
| 解放タイミング非決定的 | **カウント 0 の瞬間に決定的に解放** |
| 所有モデルは型で表せない | 所有モデルを **型で明示** |

| 項目 | C# | C++ |
|---|---|---|
| メモリ管理 | GC（自動・非決定的） | スマートポインタ（自動・決定的） |
| `new` の戻り値 | 参照（GC管理） | 生ポインタ → 即 `unique_ptr` などに包む |
| 単独所有を表す手段 | なし（言語的に表現できない） | `std::unique_ptr` |
| 共有を表す手段 | デフォルト | `std::shared_ptr`（明示的に選ぶ） |
| 確定解放 | `IDisposable` パターン | デストラクタで自動 |
| `auto q = p;` でコピー | OK（参照コピー） | `unique_ptr` は不可、`shared_ptr` は OK |

C# は **すべての参照型が同じモデル**（GC + 共有）。
C++ は **所有モデルを型で選ぶ**（unique / shared / weak）→ コードを読むだけで設計意図が分かる。

## 補足

### スマートポインタは「生ポインタを包んだクラス」

```cpp
template<typename T>
class unique_ptr {
    T* ptr;   // 中身は生ポインタ1本
public:
    ~unique_ptr() { delete ptr; }   // デストラクタで自動解放
    T& operator*() const { return *ptr; }
    T* operator->() const { return ptr; }
};
```

「魔法」ではなく、デストラクタで `delete` するクラスを `T` ごとにテンプレート化したもの。RAII の代表例。

### ヒープにオブジェクトを作るときの選択肢

| 書き方 | 解放 | 推奨度 |
|---|---|---|
| `T* p = new T();` | 手動 `delete` | ❌ レガシー、漏れやすい |
| `std::unique_ptr<T>(new T())` | 自動 | △ 例外安全に穴 |
| `std::make_unique<T>()` | 自動 | ✅ 推奨 |
| `std::make_shared<T>()` | 自動（カウント） | ✅ 共有が必要なら |

### `unique_ptr` をメンバに持つと自動的に MoveOnly に

```cpp
class Resource {
    std::unique_ptr<int> p;
    // → このクラスもコピー禁止になる
    //   （unique_ptr のコピーが delete されているため、
    //    自動生成のコピーコンストラクタも delete 扱いになる）
};

Resource a;
Resource b = a;             // ❌ コンパイルエラー
Resource c = std::move(a);  // ✅ ムーブはできる
```

`unique_ptr` を1つメンバに置くだけで、Rule of Zero で「コピー禁止＋ムーブ可」が手に入る。

### `shared_ptr` のサイズが大きい理由

```cpp
sizeof(int*);                       // 8 バイト（64bit環境）
sizeof(std::unique_ptr<int>);       // 8 バイト
sizeof(std::shared_ptr<int>);       // 16 バイト（ポインタ2本分）
```

`shared_ptr` は内部に：
- データへのポインタ
- 制御ブロック（カウンタ）へのポインタ

の2本を持つので2倍のサイズ。多用するとメモリ帯域も食う。

### スレッド安全性の注意

`shared_ptr` のカウント操作は **アトミック**（スレッドセーフ）。
ただし「指しているオブジェクトの中身」はスレッドセーフではない。

```cpp
std::shared_ptr<int> p = std::make_shared<int>(0);

// スレッドA, B でカウント増減 → OK
auto pA = p;   // 別スレッドでも安全
auto pB = p;

// しかし *p の値の操作は、自分で同期する必要がある
*p = 10;       // ← 別スレッドからのアクセスがあれば mutex 等で守る
```

### 循環参照のもう少し詳しい例

```cpp
struct Parent;
struct Child;

struct Parent {
    std::shared_ptr<Child> child;
};

struct Child {
    std::weak_ptr<Parent> parent;   // ← shared_ptr にすると循環でリーク
};
```

「親が子を所有」「子は親を弱参照」のように **所有方向** を整理すると `weak_ptr` の使いどころが明確になる。

### `enable_shared_from_this`（応用）

メンバ関数の中で「自分自身の `shared_ptr`」が欲しいときに使う特殊基底クラス。

```cpp
class MyClass : public std::enable_shared_from_this<MyClass> {
public:
    std::shared_ptr<MyClass> get_self() {
        return shared_from_this();   // 自分の shared_ptr を返す
    }
};
```

`this` から直接 `shared_ptr<MyClass>(this)` を作るとカウントが2系統になり破綻するため、これを使う。

### スマートポインタの選び方フローチャート

```
ヒープにオブジェクトを置きたい
   │
   ├─ 1人だけ所有？ ─────→ std::unique_ptr
   │
   ├─ 複数で共有？ ──────→ std::shared_ptr
   │
   └─ 共有しているが
      所有はしたくない ──→ std::weak_ptr
```
